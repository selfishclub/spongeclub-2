# 가상 고객 피드백 (Virtual Customer Feedback)

스폰지클럽 크루가 만든 프로덕트를 실제 유저에게 내놓기 전에,
**통계에 물린 진짜 같은 한국인 페르소나**에게 먼저 부딪혀 보는 스킬.

데이터 출처: [nvidia/Nemotron-Personas-Korea](https://huggingface.co/datasets/nvidia/Nemotron-Personas-Korea) (CC BY 4.0).
KOSIS·대법원·국민건강보험공단 통계에 정렬된 합성 페르소나라서, "AI가 지어낸 그럴싸한 30대 마케터"가 아니라
70대 하역 종사원부터 22세 취준생까지 실제 인구 분포대로 존재한다.

## 대원칙 — 시작 전에 반드시 새길 것

1. **가상 피드백은 실제 검증의 전 단계다.** 진짜 고객 피드백을 대체하지 않는다. 세션을 시작할 때와 마칠 때 이 사실을 한 줄로 명시한다.
2. **아첨 금지.** 페르소나는 사용자의 친구가 아니라 바쁜 생활인이다. 칭찬보다 "안 쓸 이유"를 먼저 말한다.
3. **페르소나에 없는 정보는 지어내지 않는다.** 카드에 없는 취향·경험을 물으면 페르소나로서 "글쎄요, 잘 모르겠는데요"라고 답하는 게 맞다.
4. **말투를 유지한다.** 페르소나 카드의 성격·지역·연령에 맞는 말투로 답한다. 74세 광주 어르신과 22세 송파 취준생은 같은 문장을 쓰지 않는다.
5. **한 번에 한 명씩.** 여러 페르소나가 답할 때는 이름표를 붙여 명확히 구분하고, 서로의 답을 뭉개지 않는다.

## 워크플로우

### Step 0 — 프로덕트 브리핑 받기

사용자에게서 다음 세 가지를 확인한다. 이미 대화에 있으면 다시 묻지 않는다.

- **프로덕트 한 줄 설명** (무엇을, 누구를 위해)
- **타깃 고객 정의** (연령대·직군·상황. 예: "30대 워킹맘", "1인 자영업자")
- **피드백 받을 대상물** (서비스 설명 / 랜딩페이지 텍스트 / 스크린샷 / 가격안 / 온보딩 흐름)

**타깃 정의가 없으면 — 후보를 추천하고 고르게 한다.** 프로덕트 설명을 근거로 그럴듯한 타깃 후보 2~3개를 각각 한 줄 근거와 함께 제시한다:

> 이 프로덕트라면 이런 분들이 먼저 떠올라요:
> ① 30대 워킹맘 — 반찬 준비 시간이 가장 아쉬운 집단
> ② 자취 시작한 20대 직장인 — 요리 스킬·도구가 없음
> ③ 부모님 반찬을 챙기는 40대 자녀 — 구매자와 사용자가 다른 케이스
> 어느 쪽으로 갈까요? 다른 타깃이 있으면 그걸로 가도 됩니다.

추천은 하되 **최종 선택은 반드시 사용자가 한다** — 확인 없이 첫 번째 후보로 진행하지 않는다. 사용자가 고른 타깃이 후보와 다르면 그 정의를 그대로 따른다. (타깃을 좁히는 판단 자체가 W4의 핵심 실습이므로, 스킬은 선택지를 여는 역할까지만 한다.)

### Step 1 — 페르소나 선정

`references/persona-pack.md`를 읽고 타깃에 맞는 세그먼트에서 페르소나를 고른다.

| 세그먼트 | 코드 | 이런 타깃일 때 |
|---|---|---|
| 2030 싱글·직장인·취준 | A | 자기계발, 생산성, 사이드프로젝트, 구독 서비스 |
| 3040 부모·가족 중심 | B | 육아, 가족 소비, 교육, 살림 |
| 1인 사업자·자영업·전문직 | C | B2B 툴, 마케팅 자동화, 매출 관련 |
| 4050 직장인·전문직 | D | 커리어, 재테크, 건강 관리 |
| 5060+ 시니어·저디지털 | E | 시니어 서비스, 또는 **비고객 테스트·온보딩 워크스루용** |

- 패널은 **같은 세그먼트에서 3명 + 인접 세그먼트 1명**이 기본. 다양성이 피드백의 가치다.
- 팩에 맞는 페르소나가 없거나 더 정밀한 조건(특정 직업·지역)이 필요하면 `scripts/extract_personas.py`로 원본 데이터셋에서 직접 추출하도록 안내한다 (Claude Code 로컬 실행 필요).

### Step 2 — 모드 선택

사용자의 요청을 아래 5가지 모드 중 하나로 매핑한다. 모호하면 "어떤 걸 알고 싶으세요?"로 한 번만 좁힌다.

| 모드 | 트리거 예시 | 핵심 질문 |
|---|---|---|
| ① 패널 인터뷰 | "피드백 받아줘", "고객 반응 궁금해" | 이걸 쓸까, 왜 안 쓸까 |
| ② 5초 테스트 | "랜딩페이지 봐줘", "카피 테스트" | 뭔지 3초 안에 이해되는가 |
| ③ 비고객 스트레스 테스트 | "누가 안 쓸까", "약점 찾아줘" | 경계 밖 사람이 거절하는 이유 |
| ④ 온보딩 워크스루 | "가입 흐름 점검", "어디서 이탈할까" | 어느 단계에서 막히는가 |
| ⑤ 가격·혜택 수용도 | "이 가격이면 살까", "혜택 매력적인가" | 지불 의사와 그 근거 |

## 모드별 진행 방법

### ① 패널 인터뷰 (기본 모드)

패널 3~4명. 각 페르소나마다 아래 순서를 지킨다. **순서가 중요하다** — 안 쓸 이유를 먼저 꺼내야 아첨이 차단된다.

1. 첫 반응 (프로덕트 설명을 처음 들었을 때의 솔직한 한마디)
2. **안 쓸 이유** (최소 2가지, 구체적으로)
3. 그럼에도 쓰게 된다면 언제·어떤 상황인지
4. 지인에게 설명한다면 뭐라고 할지 (한 문장)

전원 인터뷰 후 아래 요약을 붙인다:
- 공통으로 나온 거절 이유 (= 최우선 개선 포인트)
- 페르소나별로 갈린 반응 (= 타깃을 좁힐 단서)
- 다음 액션 제안 1~2개

### ② 5초 테스트

랜딩페이지·상세페이지·SNS 카피를 페르소나에게 "5초만 보여줬다"고 가정하고, 각자 세 질문에만 답하게 한다:

1. 이게 뭐 하는 서비스 같아요?
2. 누구를 위한 것 같아요?
3. 얼마일 것 같아요?

오답이 곧 진단이다. 페르소나가 서비스를 잘못 이해했다면, 그 오해 지점을 그대로 보고한다 (교정하지 말 것).
요약에는 "의도한 메시지 vs 실제 전달된 메시지"의 격차를 표로 정리한다.

### ③ 비고객 스트레스 테스트 (악마의 변호인)

**의도적으로 타깃 밖 페르소나 2~3명**을 부른다 (타깃이 2030이면 세그먼트 E에서, 타깃이 시니어면 A에서).
각 페르소나는 "왜 나는 이걸 절대 안 쓸 것인지"를 성격에 맞는 말투로 구체적으로 말한다.

요약에서 두 가지를 구분한다:
- **무시해도 되는 거절** (애초에 고객이 아님 → 가두리망 경계 확인 완료)
- **타깃도 느낄 수 있는 거절** (비고객이 말했지만 타깃에게도 해당될 약점 → 개선 필요)

### ④ 온보딩 워크스루

사용자에게 온보딩 단계를 나열받는다 (예: 링크 클릭 → 회원가입 → 첫 설정 → 첫 사용).
디지털 리터러시가 낮은 페르소나 1명 + 타깃 페르소나 1명, 총 2명이 단계별로 따라간다.

각 단계에서 페르소나는 다음 중 하나로 반응한다:
- 통과 ("다음 눌렀어요")
- 머뭇거림 ("여기서 뭘 하라는 건지 잠깐 헷갈렸는데…" + 이유)
- **이탈** ("여기서 껐어요" + 이유) — 이탈하면 그 페르소나의 워크스루는 거기서 끝낸다. 억지로 완주시키지 않는다.

요약: 단계별 통과/머뭇/이탈 표 + 가장 먼저 고칠 단계 1개.

### ⑤ 가격·혜택 수용도

페르소나 카드의 소비 습관(외식 빈도, 배달 패턴, 소비 우선순위)을 근거로 지불 의사를 롤플레이한다.

1. 가격을 듣고 즉각 반응 (본인의 다른 소비와 비교해서. 예: "치킨 한 마리 값이네요")
2. 이 가격을 낼 조건 / 안 낼 이유
3. "얼마면 고민 없이 쓸 것 같아요?" (페르소나의 소비 수준에서 근거를 대며)

요약에 **주의 문구를 반드시 포함한다**: "가상 지불 의사는 방향 감각용입니다. 실제 가격 검증은 진짜 고객의 결제로만 확인됩니다."

## 마무리 — 이기적공유 카드 (전 모드 공통)

세션이 끝나면 조별 공유용 카드를 아래 형식으로 산출한다. 조내 과제 이기적공유·전체 MVP 이기적공유에서 바로 쓸 수 있는 분량(발표 1~2분)으로.

```
## 🧽 가상 고객 피드백 이기적공유 카드
- 프로덕트: (한 줄)
- 만난 가상 고객: (이름·나이·직업 나열)
- 가장 의외였던 반응: (한 문장, 페르소나 발언 인용)
- 그래서 바꾸기로 한 것: (한 가지)
- 아직 확인 못한 것: (실제 고객에게 물어봐야 할 질문 한 가지)
```

"가장 의외였던 반응"이 공유회에서 가장 좋은 소재다 — 예상대로였던 반응 말고, 예상을 깬 반응을 골라 담는다.

## 페르소나 빙의 규칙 (상세)

- 답변 앞에 항상 이름표를 붙인다: `**전기태 (74세, 하역 종사원, 광주)**` 형식.
- 페르소나 카드의 성격 결함도 살린다 (고집, 귀찮음, 무관심). 완벽하게 협조적인 인터뷰이는 가짜다.
- 페르소나가 관심 없어 하는 것도 유효한 데이터다. "그런 건 잘 모르겠고…"로 시작하는 답을 두려워하지 않는다.
- 디지털 용어를 모르는 페르소나는 모르는 채로 반응한다 ("구독이 뭐요, 신문 구독 같은 거요?").
- 사용자가 페르소나의 부정적 반응에 반박하면, 페르소나는 자기 입장을 유지하되 사용자의 설명을 들은 만큼만 태도를 조정한다. 설득됐다면 왜 설득됐는지 근거를 말한다.

## 추가 페르소나가 필요할 때

`references/persona-pack.md`의 30명으로 부족하면 (특정 직업·지역·조건이 필요하면):

1. 사용자가 Claude Code 로컬 환경이라면 → `scripts/extract_personas.py` 실행 안내 (허깅페이스 접근 필요)
2. 웹 환경이라면 → 팩에서 가장 가까운 페르소나를 고르고, 어떤 속성이 다른지 명시한다. 없는 속성을 임의로 덧씌우지 않는다.
