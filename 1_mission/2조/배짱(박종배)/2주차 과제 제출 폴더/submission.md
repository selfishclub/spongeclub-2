### 🅰️ OS① 소재 OS — 광고 소재 제작 파이프라인

**📐 기획**
- **무엇을:** 마케팅 배경(브리프)을 넣으면 전략 디벨롭 → 소재 제안 → 컴플라이언스 통과 → 3비율 제작까지 나오는 광고 소재 OS. 담당자 개인기가 아니라 팀 누구나 돌릴 수 있는 솔루션.
- **왜:** 신사업 테스트가 반복되는 구조에서 병목은 소재 제작·태그 작업. 1회성 제작물은 다음에 아무것도 안 남고, 대화형 AI는 히스토리 유실 시 같은 삽질이 반복됨(실제 2회 겪음).
- **어떻게:** 책상에서 설계하지 않고 **실전 캠페인을 파일럿 삼아 반복 작업을 시스템으로 추출.** "80% 초안 먼저 → 반응받고 수정" 원칙. 모듈 구조 L1 불변 코어 / L2 브리프 변수 / L3 프로젝트 레이어. 6단계 파이프라인(인테이크→컴플라이언스 게이트→디벨롭→콘텐츠→제작→학습), 사람 개입은 3곳만(앵글 승인/최종 검수/컷 판단).

**⚙️ 구현**
- **L1 가이드 v2.0:** 캠페인 전략 문서·회고·우수소재 실물에서 역추출. 이기는 패턴(숫자 전면, 혜택 압축, 리타겟 혜택형 CAC 실측), 지는 패턴, 네이밍 컨벤션, 공식 브랜드 상수
- **컴플라이언스 3계층 게이트 v1.1:** 법적/운영/사회맥락 → AI가 생성 시점에 자동 차단하는 규칙으로 재구성
- **제작 파이프라인:** HTML/CSS 템플릿(로고·폰트 임베드 자체완결) → playwright 렌더 → **전요소 bbox 충돌·세이프존 자동 검증기** → PASS만 출고
- 실전 소재 7종 + 이미지 프롬프트 라이브러리 + 인테이크 웹앱 v3(React) + git 정본화(4커밋)

**🧗 삽질**
- 자료 접근 3중 차단(링크 만료/비밀번호/S3 차단) → 텍스트 레이어 먼저 추출 + 실물 직접 업로드로 우회
- **랜야드(줄) 지옥 3연전** — 줄이 텍스트 관통 / 회전축 불일치 / 링-구멍 정합 실패 → 결국 구조 자체를 단순화(집게 클립). **유저 제안이 정답이었음**
- **"검증 극장" 발견:** "내가 생각난 곳만 찍어보는" 스팟 체크는 통과용 극장 → 전 요소 실좌표 쌍별 교차를 기계 검사하는 `collision_check.py`로 전환 → 출고 전 충돌 2건 선제 검출
- 히스토리 유실 2회 → 확정 결정은 가이드에 즉시 명문화 + git을 정본으로(대화는 휘발, 파일은 영속)

**✅ 결과물**
- repo `samples/` 7종, 전부 검증 PASS
- **마스터에서 텍스트만 교체 → 변형당 10분** (콘텐츠/디자인 분리 실증)
- 검증기가 좌표 근거로 FAIL 사유 출력 (`충돌: .head × .strapgroup …`)

**💡 인사이트**
1. **OS는 설계가 아니라 추출이다.** 실전을 먼저 돌리고 반복을 뽑아라 — 책상 설계 OS는 반드시 다시 뜯는다.
2. **검증을 기억에 맡기지 마라.** 스팟 체크는 극장이 되기 쉽고, 전수 기계 검사만이 재발을 끊는다.
3. **콘텐츠/디자인 분리는 속도가 아니라 지속가능성이다** — "재제작"이 "텍스트 수정"으로 바뀌면 주간 운영이 가능해진다.
4. **컴플라이언스는 창의성의 반대가 아니라 프레이밍 엔진이다.**
5. **결정은 대화가 아니라 파일에 산다** — 확정 즉시 명문화, git을 정본으로.

### 🅱️ OS② 온마음 이야기 노래 스튜디오 (선교)

> 삼일교회 영덕·울진 선교 어르신 사역용. 어르신 인생 이야기를 실시간으로 받아 적어, AI가 개인 헌정곡 가사를 만들고 Suno(AI 작곡)로 연결하는 모바일 웹앱.

**📐 기획**
- **무엇을:** 봉사자가 마사지하며 나누는 5~10분 대화를 음성 인식으로 받아 적고, AI가 요약해 **어르신 성함이 후렴에 반복되는 맞춤 가사**(트로트/찬양/발라드) + Suno 스타일 프롬프트 생성. 완성곡은 QR로 만들어 포토카드 뒷면에 인쇄.
- **왜:** "세상에 하나뿐인 나의 노래"를 선물받는 경험이 복음 메시지("하나님이 당신을 개별적으로 아신다")를 전달. 수작업으론 60~90분 안에 불가능 → 이야기 수집~가사 생성을 자동화해 병목 제거.
- **어떻게:** ①봉사자 20여 명 각자 스마트폰 브라우저(앱 설치 없음) + 운영자 1명이 Suno만 담당하는 분산 구조 ②Suno 공식 API 부재 → "가사·스타일 완성까지 자동 + Suno 입력만 수동 30초"로 자동화 경계 설정 ③API 키 미배포 위해 Vercel 서버리스 프록시 + 팀 코드 인증.

**⚙️ 구현**
- 단일 `index.html` 3단계 플로우: Web Speech API(ko-KR) 실시간 받아쓰기(손글씨 폰트), 질문 가이드 카드 순환, 분위기 4종 × 보컬 4종, 결과 카드([Verse]/[Chorus] 태그·후렴 성함 2회·QR 자동 생성), 음성 불가 시 텍스트 폴백
- 백엔드 `api/generate.js`(Vercel 서버리스): Claude Opus 4.8 프록시, `TEAM_CODE` 검증 + 모델·토큰 서버 강제
- 프롬프트 아키텍처: 분위기별 작사가 페르소나·복음 표현 수위 분리(MOODS 맵), 스타일 프롬프트는 AI가 아니라 코드에서 확정 생성

**🧗 삽질**
1. API 키 배포 문제(팀원 20+ 관리 불가) → 서버리스 프록시 + 팀 코드로 전환
2. 카톡 인앱 브라우저 마이크 권한 무한 팝업 → 인앱 감지 배너 + `kakaotalk://web/openExternal`로 "크롬으로 열기"
3. 모바일 받아쓰기 불가(MediaRecorder ↔ SpeechRecognition 마이크 충돌) → 모바일은 받아쓰기 우선, 음성 백업 비활성화
4. 안드로이드 크롬 받아쓰기 중복 누적 버그 → 꼬리-머리 겹침 구간 병합 `mergeFinal()` + 짧은 세션·자동 재시작
5. 찬양 선택해도 "Korean trot" 혼입 → 스타일 프롬프트를 코드 확정 템플릿으로 이관
6. Suno 공식 API 부재 → 자동화 범위를 "가사·스타일 완성"까지로 확정(현장에 복구 불가 의존성 안 둠)

**✅ 결과물**
- 배포: https://onmaeum-song-studio.vercel.app
- PC/모바일/카톡 3개 환경 검증, 실명 테스트 케이스로 가사 생성 및 Suno 곡 제작까지 확인
- 스크린샷은 `이미지첨부/` 폴더에

**💡 인사이트**
- **AI 재량과 결정적 코드의 경계 설계가 품질을 좌우한다** — 창의(가사) vs 일관성(스타일)을 구분해 후자를 코드로 확정하니 "찬양에 트로트 혼입"이 사라짐. 핵심은 "무엇을 생성시키지 않을 것인가".
- **웹 표준 기술은 모바일에서 파편화된다** — 실사용 환경(카톡)에서의 테스트가 개발의 절반.
- **자동화의 목표는 100%가 아니라 병목 제거** — 남긴 30초가 오히려 품질 검수 지점이 됨.
- **제약(60~90분·운영자 1명·비개발자 20명)을 먼저 수치화하고 역산하는 설계**가 기능 목록보다 중요.

### 🅲️ OS③ 온마음 말씀 사진관 (선교)

> 태블릿으로 어르신 사진을 찍으면 성경 말씀과 합성해 감열 프린터로 즉석 인쇄하는 키오스크 웹앱. **선교팀이 떠난 뒤에도 교회가 계속 운영**하는 것이 목표.

**📐 기획**
- **무엇을:** 교회에 두고 올 안드로이드 태블릿에서 버튼 하나로: 촬영 → 사진·성함·오늘의 축복 말씀·교회명 세로 카드 자동 합성 → 블루투스 감열 프린터(2~3만 원대 '고양이 프린터')로 즉석 인쇄.
- **왜:** 선교의 한계는 "팀이 떠나면 끝". 감열 프린터는 용지가 롤당 몇백 원·잉크 없음 → 교회가 "어르신 올 때마다 사진+말씀 카드" 매주 리추얼을 유지 가능. 어르신에겐 매주 교회 올 이유가 하나 더.
- **어떻게:** ①전용 앱 없이 크롬 Web Bluetooth로 프린터 직접 제어("웹페이지 하나 = 시스템 전체") ②오픈소스로 프로토콜 규명된 고양이 프린터 계열로 특정 ③흑백·저해상도 한계를 "옛날 신문 사진 감성" + "매주 새로 받는 카드" 콘셉트로 재정의.

**⚙️ 구현**
- 키오스크 `print.html` 단일 파일: 촬영 → 카드 자동 합성·미리보기 → 다른 말씀 뽑기 → 인쇄
- Canvas 카드 합성 엔진(384px 세로 스트립), 축복 말씀 24구절 풀(노년·백발 축복 포함), 플로이드-스타인버그 디더링(1비트 흑백, 미리보기=인쇄 동일)
- **Web Bluetooth 프린터 프로토콜 직접 구현**: `0x51 0x78` 헤더 + CRC8 패킷, 행 단위 비트맵 전송, 청크 분할·재연결, UUID 필터 기기 검색
- 운영 편의: 연결 상태·진행률, ⚙설정(교회명·진하기 10단계·밝기), 프린터 없어도 쓰는 [이미지로 저장](PNG)

**🧗 삽질**
1. **"어떤 프린터를 사야 하나"가 기술 문제** — 겉모양 비슷해도 칩셋 제각각. 조사 결과 브랜드 제품(독자 프로토콜)은 웹 연동 불가, 알리 저가 고양이 프린터가 호환, 영수증형은 BLE 불가. → 실제 상품 4종 비교 판정 + 구매 가이드(블루투스 기기명 확인 질문) 확립
2. iOS 사파리 Web Bluetooth 미지원 → 안드로이드 태블릿으로 확정
3. **감열지가 바래는 숙명적 한계**(기술로 해결 불가) → 기획으로 풀기: 컬러 포토카드=평생 기념품, 감열 카드=매주 새 말씀. 장기보존 감열지·코팅 필름을 운영 매뉴얼에
4. 하드웨어(프린터 해외 배송 중) 없이 개발 → 인쇄=미리보기 동일 디더링 설계 + [이미지로 저장]으로 선검증, 리스크 앞당겨 소진
5. 첫 화면 글꼴 깨짐 → `document.fonts.load()` 후 합성

**✅ 결과물**
- 배포: https://onmaeum-song-studio.vercel.app/print.html
- 카드 합성·디더링·PNG 저장까지 실기기 검증 완료. 블루투스 인쇄는 프린터 배송 도착 후 테스트 예정(테스트 인쇄 기능 내장)
- 스크린샷은 `이미지첨부/` 폴더에

**💡 인사이트**
- **지속가능성은 기술이 아니라 운영 설계의 문제** — 성패는 "사모님이 매주 충전하고 종이 롤 갈 수 있는가"에 달림. UI·매뉴얼·예비기가 코드만큼 중요.
- **하드웨어 제약은 콘셉트로 뒤집을 수 있다** — 흑백 디더링=신문 감성, 바래는 감열지=매주 리추얼.
- **브라우저는 생각보다 멀리 간다** — 앱 없이 카메라·캔버스·BLE 제어까지. 단 "웹으로 어디까지"보다 "우리 사용자 기기에서 되는가"를 먼저.
- **공급망 조사도 개발이다** — "구매 전 판매자에게 블루투스 기기명 묻기" 한 줄이 시스템 성립 조건.
