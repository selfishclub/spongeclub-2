# 1주차 — 나만의 OS 만들기

## 🎯 미션 1. 내 OS 재료 찾기

- **과정 (어떻게 찾았나):**
  인터뷰 스킬 5스텝을 밟았다. 반복 업무는 이미 자동화해놨는데 거기서 더 발전이 없다는 정체감이 출발점이었다(S1~S2) — 회사 안에서 더 파고들지, 이직할지 갈림길에서 AI로 산업이 재편되는 걸 보고 AI 회사들을 찾아보기 시작했는데, 회사가 밸류체인 어디(모델/파인튜닝/애플리케이션) 있는지·어떤 섹터인지 파악하는 데 시간이 오래 걸리는 게 진짜 마찰이었다(S2). 일 밖으로 나가보니 '기록·공부'를 놓치고 있었지만, 이걸 커리어 방향과 얼라인시키고 싶다는 마음이 더 컸다(S3). 회사 리서치 vs 기록·공부 중 무게를 재보니 — 기록은 어차피 클로드랑 대화하면서 자연히 쌓이지만, 리서치는 매번 손이 많이 가서 리서치 쪽을 골랐다(S4).

- **결과:** 🟡 내 OS 재료 카드

  | 항목 | 내용 |
  |---|---|
  | 영역 | 일 (AI 회사 리서치 · 커리어) |
  | 걸리는 지점 | 반복 업무는 이미 자동화했는데 정체감을 느낌 → AI 회사들이 밸류체인 어디(모델/파인튜닝/애플리케이션)에 있고 어떤 섹터를 차지하는지 파악하는 데 시간이 오래 걸림 |
  | 지금은 | 크롤링 + API/MCP로 회사 정보를 모아 아침마다 데일리 리포트로 받아보는 것까지 만들어봤지만, 회사는 매일 크게 안 변하고 공개된 기술 정보도 적어서 한계 → 결국 커피챗·지원 같은 직접 액션으로 사람을 만나야 판단이 선다는 걸 깨달음 |
  | OS가 된다면 | 회사에서 시간이 붕 뜰 때 자연스럽게 붙는 게이트웨이 — 1차 스크리닝(핏 판단)부터 시작해 더 많은 회사 기회를 훑고, 판단 안 되는 부분은 커피챗·지원 같은 액션으로 넘겨줌 |
  | 한 문장 | "붕 뜨는 시간이, 뿌듯함과 만족감이 남는 시간이 된다" |
  | 첫 한 걸음 | 이력서/프로필 넣고 회사 검색하면 리서치 → 핏 스코어 → 액션(커피챗/지원)까지 한 번에 나오는 웹앱 — Career OS |

- **느낀 점:**
  이미 자동화를 다 했다고 생각했는데, 크롤링·API·데일리 리포트까지 만들어보고 나서야 진짜 마찰이 뭔지 보였다 — 정보가 부족한 게 아니라, 판단을 내리려면 결국 사람을 직접 만나야 한다는 것. 그래서 자동화를 더 세게 미는 대신, 판단이 안 되는 지점에서 커피챗·지원 같은 액션으로 자연스럽게 넘어가는 다리를 만드는 쪽으로 방향을 틀었다. "나는 이상하게 ___가 거슬려"를 따라가 보니, 핵심 마찰은 시간이 *없어서*가 아니라 *내 것이 아니라서* 괴로운 것 = **주도권의 부재**였고, 내가 원하는 한 조각은 "나를 위한 커피챗 — 내가 먼저, 내가 원해서"였다.

## 💡 미션 2. 내 OS 기획

**커리어 OS — 회사 타겟 모드**
가고 싶은 회사가 생겼을 때, **리서치 → 핏 점수 → 액션(커피챗 or 지원)** 까지 한 흐름으로 끌어주는 OS.

- 표면 목적: "뭘 어떻게 해야 할지 몰라" 멈추는 순간을 없애기
- 숨은 목적: 어디에 끌렸는지가 쌓여 내 방향이 드러난다
- 구조: ①회사 리서치(5앵글) ②JD 핏 스코어링 ③유사 기업 + 산업 트렌드 ④커피챗 or 지원 액션 분기

**왜 이게 나인가:**
BD 경력의 무기인 "연결·명분 생성"을 그동안 회사를 위해서만 썼다. 이 OS는 그 능력을 처음으로 나 자신에게 쓰게 한다.

## ⚙️ 미션 3. 내 OS 구현

Next.js 웹앱 + Claude Code 스킬로 구현.

- **웹앱**: 이력서/PDF/스크린샷 입력 → 회사·사람 검색+확인 → 리서치·핏점수·액션플랜(또는 커피챗 액션) 자동 생성
- **스킬파일**: `커피챗-os/SKILL.md`

**핵심 설계:**
- 이력서 먼저 → 회사 검색 순서 (내가 누구인지 파악이 선행)
- 채용 URL 붙여넣으면 AI가 1차 소스로 사용 (할루시네이션 방지)
- STEP별 카드로 분리 출력 (리서치 / 핏점수 / 유사기업 / 액션)
- 검색·확인 단계 추가: 이름만 치면 실제 웹검색으로 후보를 찾아서 보여주고 내가 고름 (회사/사람 정확도, 동명이인 문제 해결)
- 회사 vs 사람 모드 분기: 사람(커피챗 대상)이면 4스텝이 아니라 리서치+액션 2스텝만
- 프로필 입력 화면에 PDF/이미지 첨부 → 텍스트 자동 추출해서 붙여넣기 대체
- 검색 히스토리 자동 누적: 분석 끝나면 자동 저장, 헤더 "히스토리" 버튼으로 언제든 조회, 클릭 한 번으로 재분석
- 히스토리 패턴 분석: 핏 점수 있는 검색이 5개 이상 쌓이면 "📊 패턴 보기" 버튼이 뜨고, 지금까지 본 회사들을 AI가 스스로 카테고리로 묶어 카테고리별 평균 핏 점수 + 왜 그런 차이가 나는지 인사이트를 뽑아줌. 혼자 비교하지 않아도 "내 경력이 어떤 도메인에 더 잘 맞는지"가 쌓일수록 자동으로 보이는 구조

**삽질 과정:**
- 처음엔 하고 싶은 거 10개를 다 쏟아놓고 시작함(레쥬매 넣기, JD 스크리닝, 핏 점수화, 유사기업, 트렌드, 커피챗 명분까지 전부). 브레인스토밍하면서 "일단 1단계는 회사 타겟 모드까지만"으로 범위를 잘라냄.
- 1차로 만들고 써보니까 바로 회사 검색부터 들어가더라. "어? 근데 내가 누구인지도 모르는데 이게 되나?" 싶어서 이력서/프로필부터 받는 순서로 뒤집음.
- 회사 이름이 좀 낯선 거(예: tynapse) 넣었더니 AI가 지 맘대로 "시냅스"로 착각해서 딴 얘기를 지어냄. 이거 보고 소름 돋아서 — 회사 정보는 내가 직접 붙여넣으면 그걸 1차 소스로 무조건 쓰게 룰 박음. 안 그러면 그럴싸하게 헛소리하는 게 제일 위험하더라.
- 결과 나오는 동안 화면이 계속 아래로 밀려서 위에 내용을 못 읽는 것도 짜증나서 고침. 한 번에 다 쏟아내지 말고 STEP별로 카드처럼 끊어서 보여주게 바꿈.
- vercel에 배포하는데 프로젝트 이름 `Career_OS`가 대문자·언더바 써서 거부당함 — 소문자로 다시. API 키 환경변수도 Production에만 등록해놓고 왜 안되지 하다가 다시 등록.
- 파일도 처음엔 profile.md, README.md, SKILL.md, submission.md 이렇게 4개까지 늘어났었는데, "어차피 웹앱에 경력 쓰는 칸 있는데 이걸 왜 파일로 또 만들었지" 싶어서 SKILL.md 하나로 정리함. 개인정보 정리도 3~4번은 다시 훑었음.
- 근데 곰곰 생각해보니 애초에 AI가 실시간 웹검색 없이 그냥 "아는 척"으로 리서치하고 있었던 거였음. 그래서 검색 도구를 진짜로 붙이고, 이름 치면 후보 목록 보여준 다음 내가 고르는 확인 단계를 새로 만들었음 — 할루시네이션·동명이인 문제가 한 방에 정리됨.
- 사람(커피챗 대상)을 검색했는데 회사용 4스텝 카드가 그대로 나와서 어색했음. "핏 점수·유사기업"은 사람한텐 말이 안 되니까, 사람이면 리서치+액션 2스텝만 나오게 분기침.
- 정보가 거의 없는 사람 이름으로 테스트하다가 AI가 STEP1만 쓰고 "정보 더 주세요"라고 되묻고 멈추는 걸 발견 — 근데 매번 그러진 않고 가끔만 그럼(비결정적). 5번 연속 돌려서 재현 확인하고, "정보 없어도 STEP2까지는 무조건 채워라"를 프롬프트에 못 박아서 고침.
- worktree 정리하다가 실수로 로컬 `.env.local`을 지워서 API 키가 통째로 날아감. Vercel에 저장해뒀으니 괜찮을 줄 알았는데, "Sensitive" 타입으로 등록한 값은 한 번 설정하면 그 이후로 CLI로도 절대 다시 못 읽어온다는 걸 뒤늦게 알았음(`vercel env pull` 해도 빈 문자열만 나옴) — 결국 키 새로 발급받아서 로컬·Vercel 둘 다 다시 등록.
- PDF/이미지 첨부 기능은 반대로 순조로웠음. 처음부터 검증 로직을 순수 함수로 떼어내서 테스트 먼저 짜놓으니, 실제 API 라우트 붙일 때 타입 에러 한 번 없이 바로 통과됨 — 앞의 삽질들 덕에 패턴이 잡힌 느낌.
- 히스토리 기능 만들면서 이 프로젝트 최초로 DB(Upstash Redis)를 붙였는데, 이게 생각보다 삽질 연속이었음. "upstash"라고만 쳤더니 "제품이 여러 개니 정확히 골라라" 거부당함 → upstash-kv로 특정. 그다음엔 마켓플레이스 약관을 브라우저에서 직접 눌러야 해서 링크 받아서 동의. 그것도 통과하고 나니 이번엔 DB가 만들어지긴 했는데 내 프로젝트엔 연결이 안 돼있어서 따로 연결 명령을 한 번 더 쳐야 했음.
- 겨우 연결하고 환경변수 받았더니 예상한 이름(UPSTASH_REDIS_REST_URL)이 아니라 KV_REST_API_URL이라는 다른 이름으로 와서 코드를 그 자리에서 고침 — Vercel이 예전 이름으로 호환되게 만들어놓은 거였음.
- 이 와중에 `vercel env pull`을 환경 지정 안 하고 치는 바람에 API 키가 또(!) 빈 값으로 덮어써짐. 전에도 한 번 겪었던 "Sensitive 타입은 한번 정하면 다신 못 읽어온다" 문제를 이번엔 두 번이나 반복해서 겪음 — 다행히 키 값을 기억해뒀다가 바로 복구. "vercel env pull은 꼭 --environment=production으로"가 이제 완전히 체화됨.
- 테스트하면서 실제 DB에 테스트 데이터(가짜 회사·사람 이름)가 남았는데, 지우려고 할 때마다 시스템이 "이건 사용자 확인 필요한 파괴적 작업"이라고 막아서 매번 직접 확인받고 지움 — 성가시긴 한데 이런 안전장치가 왜 있는지 이번에 제대로 체감함.
- 브라우저로 직접 클릭해볼 방법이 없는 환경이라(서버만 있고 화면은 없음), 대신 실제 API를 코드로 직접 호출해서 "진짜 회사 리포트 생성 → 요약 추출 → 저장 → 조회"까지 전체 흐름을 검증하는 방식으로 우회함. 클릭 테스트는 아니지만 데이터가 실제로 도는지는 확실히 확인됨.
- 여러 회사로 실제 리서치를 반복해서 돌려보다가 Anthropic 계정 크레딧이 바닥나서 "API 접근이 막혔다"는 메일을 받음. 알고 보니 크레딧은 API 키 단위가 아니라 **조직(Organization) 전체가 공용으로 쓰는 잔액**이라, 이 프로젝트에서 쓴 만큼 다른 프로젝트 크레딧도 같이 줄어드는 구조였음.
- 이 일로 지금 배포된 웹앱에 로그인·비밀번호·요청 제한이 전혀 없다는 걸 다시 확인함 — 링크만 알면 누구나 내 API 크레딧으로 무제한 호출이 가능한 상태. 이번 주엔 기능 구현에 집중하느라 아직 막아두진 못했고, 다음 단계로 비밀번호 게이트나 요청 제한을 붙여야 한다는 걸 실제 사고로 배움.
