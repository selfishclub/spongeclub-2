# 3주차 — 내 OS 최종 완성 🏁

## 🎯 미션 1. 내 삶을 돕는 OS 최종 완성

### 한 줄 요약

**3주차엔 기능을 하나도 안 붙였다. 대신 "던진 게 반드시 도착한다"를 만들었다.**
2주차 dylan-os는 라우팅이 다 됐다. 그런데 일주일 살아보니 진짜 문제는 라우팅이 아니라 **봇이 조용히 죽어 있는 시간**이었다. 결과: 7/13~7/18 수신 46건 / 처리 46건 — **유실 0.**

---

### ✅ 완성한 것 (무엇을)

2주차 제출문 마지막에 이렇게 썼었다.

> *"수집만 잘하는 세컨브레인은 조용히 새고 있어도 티가 안 난다."*

3주차는 그 문장을 끝까지 밀어붙인 주다. 새 입력도, 새 출구도 안 만들었다. **`vault/inbox.md`에 뭘 더 담을까가 아니라, 담기기 전에 사라지는 걸 어떻게 막을까**만 했다.

| # | 고친 것 | 증상 (실제로 겪은 것) |
|---|---|---|
| 1 | **wake grace sleep 롤백** | 맥 서스펜드를 넘어 유예 sleep이 늘어나서 90초가 **20~50분**이 됐다. 그동안 텔레그램 폴러가 통째로 귀머거리 |
| 2 | **yt-dlp `--no-playlist`** | `&list=`가 붙은 유튜브 URL 하나 던졌더니 **재생목록 전체를 순회**하며 영영 안 돌아옴. 봇이 멈춘 걸로 보임 |
| 3 | **좀비 세션 감지** | `claude`도 `bun`도 **살아있는데** 텔레그램 long-poll만 조용히 죽어 있음. 7/13 한 시간 통째로 무응답인데 프로세스 체크는 전부 정상 |
| 4 | **`.received.log` — 유실 안전망** | 텔레그램은 폴러가 가져가는 **순간 서버에서 지운다.** 그 뒤 데몬이 죽으면 원본이 어디에도 안 남아 재생 불가 |

**4번이 이번 주의 본체다.** 앞의 3개는 "덜 죽게" 만드는 거고, 4번은 **"죽어도 잃지 않게"** 만드는 거다. 전자는 확률을 낮추고 후자는 손실을 0으로 만든다.

### 💬 피드백 반영한 점

솔직히 말하면 조에서 코드 리뷰 형태의 코멘트를 직접 받진 못했다. 대신 **두 종류의 피드백**을 반영했다.

**1. 실사용이 준 피드백 — "던졌는데 답이 안 와"**

2주차 내내 가장 자주 겪은 게 이거다. 그런데 원인이 매번 달랐다(서스펜드 · 플레이리스트 무한루프 · 좀비 세션). **증상 하나에 원인 셋.** 그래서 원인별로 막는 대신, **원인이 뭐든 잃지 않는 층**을 하나 더 깔았다.

**2. 채널이 준 피드백 — 이건 나만의 문제가 아니었다**

질문 게시판을 보니 **텔레그램 OS를 만든 사람이 전부 같은 벽**에 서 있었다. "맥을 24시간 켜둬야 하나요?" 답은 "맥미니를 사라" 아니면 VPS인데, **VPS는 난이도가 높아 선뜻 권하기 어렵다는 데서 다들 멈춰 있었다.**

여기서 방향을 바꿨다. **하드웨어를 바꾸는 게 아니라, 죽는 걸 전제로 설계하면 된다.** 맥은 자고, 크래시 나고, 세션은 좀비가 된다 — 그걸 없애려 하지 말고 **잃지 않게** 만드는 것. 그래서 3주차 산출물은 맥미니도 VPS도 아니고 **셸 스크립트 100줄 + `CLAUDE.md` 규칙 10줄**이다.

### ⚙️ 결과물

**구조 — 안전망이 어떻게 도는가**

```
텔레그램 메시지 도착
   ↓
① 세션이 처리를 시작하기 "전에" 무조건 먼저 한 줄 기록
   vault/.received.log  ←  <message_id> | <ts> | 내용 첫 100자
   ↓
② CLAUDE.md 라우팅 규칙대로 처리 (캘린더 / todos / vault)
   ↓
③ 완전히 끝난 뒤에만 한 줄 더
   vault/.received.log  ←  DONE <message_id>

────────── 여기서 데몬이 죽으면? ──────────

래퍼가 재시작될 때마다 (첫 기동 포함) process_dropped 실행
   ↓
awk로 "받았는데 DONE이 없는 줄" 추출  =  흘린 메시지
   ↓
🔄 "답 못 한 메시지 N개 발견 — 지금 처리한다" 텔레그램 알림
   ↓
headless claude -p 로 CLAUDE.md 라우팅을 그대로 재생
   ↓
보고만 하는 게 아니라 실제로 처리되고 DONE이 찍힌다
```

**좀비 감지 — 존재가 아니라 "일한 흔적"을 본다**

```bash
# 프로세스 살아있음 체크는 이 버그를 100% 놓친다.
# claude는 뭐라도 하면 로그를 쓴다 → 로그가 20분간 안 자라면 좀비로 간주하고 재시작.
LOG_MTIME=$(stat -f %m "$LOG_FILE")
IDLE=$((NOW - LOG_MTIME))
if [ "$IDLE" -gt 1200 ]; then ... kill "$CLAUDE_PID"; break; fi
```

### 💡 알게 된 인사이트

**1. "프로세스가 살아있다"는 "일하고 있다"가 아니다.**
`ps`로 확인되는 건 프로세스의 **존재**지 **기능**이 아니다. `claude`도 `bun`도 멀쩡한데 텔레그램 long-poll만 죽어 있는 상태가 실제로 존재했고, 존재 체크는 이걸 절대 못 잡는다. → **헬스체크는 "살아있나"가 아니라 "최근에 뭘 했나"를 봐야 한다.** 로그 mtime이 가장 싸고 정확한 신호였다.

**2. 안전망은 남의 서버에 두면 안 된다.**
텔레그램은 폴러가 update를 가져가는 순간 서버에서 지운다. 즉 **"다시 받아오면 되지"가 성립하지 않는다.** 내 쪽에 수신 원장이 없으면 크래시 한 번에 영구 소실이다. 세컨브레인에서 이건 치명적인데, **잃어버린 사실 자체를 모른다**는 게 더 치명적이다. → 외부에서 들어오는 것은 **처리보다 먼저 기록**한다. 순서가 뒤바뀌면 안전망이 아니다.

**3. 🔴 안전망을 깔았더니 감시자가 과민해졌다 — 아직 안 고쳤다.**
좀비 감지를 "20분간 로그 무활동"으로 잡았는데, **한가한 봇도 로그를 안 쓴다.** 그래서 아무 일 없는 날엔 20분마다 정직하게 재시작한다. 7/18 하루 16번.

```
2026-07-18 18:00:52 zombie detected (no log activity for 1217s), restarting claude
2026-07-18 18:21:26 zombie detected (no log activity for 1209s), restarting claude
2026-07-18 18:41:58 zombie detected (no log activity for 1207s), restarting claude
```

코드 주석에 내가 "한가한 봇은 재시작해도 잃는 게 없으니 오탐은 무해하다"고 적어놨는데, **무해한 것과 옳은 것은 다르다.** 재시작마다 세션 컨텍스트가 날아가고, 진짜 좀비와 그냥 한가한 상태를 여전히 구분 못 한다. → 다음 수정은 **하트비트**다. 폴링 루프 자체가 주기적으로 "나 듣고 있다"를 찍게 하면 침묵과 죽음이 분리된다.
**"살아있음의 증거를 일감에 의존하면, 일이 없는 게 죽음으로 보인다."**

**4. 2주차의 결론이 3주차에도 그대로 유지됐다.**
2주차에 "고친 것 5개 중 4개가 코드가 아니라 `CLAUDE.md` 규칙 한두 줄"이라고 썼다. 이번 주 핵심인 유실 방지도 마찬가지다 — **"처리를 시작하기 전에 가장 먼저 `.received.log`에 한 줄을 append 한다"는 문장 하나**가 규칙의 전부고, 셸 쪽은 그걸 재생만 한다. 미들웨어를 짜지 않았다. **판단은 문서에, 복구는 스크립트에.** 이 경계가 3주 연속으로 맞았다.

**5. 이번 주의 진짜 교훈: 완성은 기능이 아니라 신뢰다.**
3주차 미션이 "최종 완성"이라 기능을 더 붙일까 고민했다. 안 붙이길 잘했다. **하루에 한 번 못 미더우면 OS가 아니라 장난감이다.** 기능이 하나 더 있는 것보다 **던진 게 반드시 도착한다**가 훨씬 비싸다.
