### 📐 기획

기존에는 줌 미팅/슬랙 인풋 → 태스크 정리 → 기획서 → Figma → 싱크 반영 → GitHub 이슈 → QA 과정을 Claude Chat의 Project 기능으로 돌리고 있었는데, 대화가 흩어지고 버전 관리가 안 되는 문제가 있었다. 이걸 폴더 하나에 자기완결적으로 담긴 OS로 만드는 것을 목표로 잡았고, 두 갈래로 구현했다.

- **결과물 A — 회사 Product Design OS**: 흩어져 있던 디자인 워크플로우를 Claude Code 기반의 로컬 파일 OS로 구축
- **결과물 B — 포토그래퍼 운영 어드민 "ADMIN"**: 1인 웨딩·커머셜 포토그래퍼의 예약·프로젝트·인보이스·일정을 한 곳에서 굴리는 실사용 어드민 대시보드

### ⚙️ 구현

#### 결과물 A — 회사 Product Design OS

"회사 Product Design OS" — 흩어져 있던 디자인 워크플로우를 Claude Code 기반의 로컬 파일 OS로 구축.

1. **시작점: 기존 워크플로우의 문서 포맷을 그대로 이식**
   - 검증된 자산(태스크 정리/기획서/이슈/QA 문서 포맷)은 새로 만들지 않고 기존 workflow-guide에서 그대로 포팅
   - Notion 중심 → 로컬 파일 중심으로 전환 결정 (버전 관리·자동화에 유리)

2. **워크플로우 하네스**
   - CLAUDE.md (OS 헌법): 폴더 맵, 태스크 상태 머신 intake → overview → spec → figma → syncing → final → issued → qa → done, 라이팅 규칙(내부 문서 한국어 / 이슈·QA 영어)
   - 스킬 6종: /intake /spec /sync /issue /qa 등 — 각 단계의 절차·포맷·마무리 규칙(TASK.md 상태 갱신, 대시보드 갱신)을 스킬 파일로 명문화
   - 태스크 1개 = GitHub 이슈 1개 원칙, inbox/(대기열)와 meetings/(미팅 원본 영구 아카이브) 분리

3. **로컬 대시보드 웹사이트**
   - 의존성 제로 Node 서버 + 단일 HTML로 구축. 파일을 실시간으로 읽어서 별도 동기화 불필요
   - 상태별 칸반 보드, 태스크 상세(기획서/이슈/QA 탭), 미팅 아카이브, 지식 베이스 브라우저

4. **Product Design AI Agent로 확장**
   - 핵심 설계: 2층 구조 — ① 지식(knowledge) 레이어 위에 ② 5개 도메인 스킬(기획/UI 디자인/UX 라이팅/크리틱/핸드오프)이 얹히는 구조
   - 지식 레이어: 디자인 시스템 JSON 스냅샷(599개 컴포넌트)에서 P0 161개 컴포넌트 문서를 스크립트로 자동 시드 생성 (이름 매핑·variants·props 자동 기입, prop 결정 트리는 추출 단계에서 채움) + 토큰/용어집 문서 변환 + 페이지 패턴 6종 골격
   - 신규 스킬 9종: 지식 구축용 /ds-scan /ds-tokens /ds-component /ds-pattern /ds-flow + 실행용 /ui-draft(소스→우리 컴포넌트 매핑→Figma 초안) /ux-copy(용어→문법→명확성 3패스) /ux-critique(7렌즈 통합 분석) /ui-critique(Figma 실측값 vs 토큰 검증)
   - 서브에이전트 5종: 지식 검색 전담(ds-librarian) + 크리틱 패널 4종(flow/policy/feasibility/state) — 필요할 때만 팬아웃하는 토큰 절약 설계
   - 문서 신뢰도 라이프사이클 seed → enriched → verified 정의, 대시보드에서 진척도 추적 가능

5. **실전 검증**
   - 실제 줌 미팅 2건을 인테이크해서 태스크 정리·싱크 노트·확인 질문 목록까지 실제 산출물 생성
   - 이식성 최종 감사: 저장소 전체에 절대경로·환경 의존 0건 → 폴더째 복사하면 다른 환경에서 그대로 동작

#### 결과물 B — 포토그래퍼 운영 어드민 "ADMIN"

1인 웨딩·커머셜 포토그래퍼의 운영 통합 어드민 대시보드 "ADMIN"을 만들었습니다.

- **배포**: Vercel (main 푸시 시 자동 배포)
- **규모**: 14개 화면, 53개 소스 파일, 약 7,800줄 (TypeScript)
- **어디서 시작했나**: Create Next App 기본 템플릿 하나에서 출발해서, 흩어져 있던 실제 운영(웨딩 예약·커머셜 프로젝트·인보이스·일정·설정)을 한 곳에서 굴리는 실사용 도구로 완성

**핵심으로 만든 것**

| 영역 | 내용 |
|---|---|
| **웨딩 파이프라인** | 문의 → 상담 → 예약확정 → 촬영/보정/납품까지 단일 상태 축으로 관리. 확정 시 자동으로 예약 관리로 이동, 예약금 파이프라인 시작 |
| **문의 자동 수집** | 구글 폼 → 응답 시트 → Google Apps Script → **Supabase 자동 동기화**. 이메일·이름으로 기존 문의에 자동 매칭 |
| **인보이스** | 이중 통화(AUD/KRW), 세션 가격표 기반 자동 계산, **PDF 다운로드**(@react-pdf/renderer, 한글 폰트 임베드) |
| **캘린더** | 촬영 일정 popover + 구글 캘린더 연동 |
| **커스터마이즈** | 촬영 세션·체크리스트·안내 메일 문구를 코드 수정 없이 편집 |
| **문의 안내 메일** | 버튼 한 번으로 고객별 제목·본문·첨부 링크가 채워진 Gmail 작성 화면 열기 |
| **인증/배포 대비** | Supabase Auth + 로그인 게이트, RLS(로그인 사용자 전용) 적용 |

**데이터 흐름 한눈에**

```
[고객] 구글 폼 제출
   ↓ (응답 시트)
[Apps Script] onFormSubmitSync 트리거
   ↓ (REST + service_role 키)
[Supabase] weddings 테이블
   ↓ (StoreProvider, 낙관적 업데이트)
[어드민 화면] 실시간 반영 (당일 문의는 NEW 뱃지)
```

**기술 스택**: Next.js 16(App Router) · TypeScript(strict) · Tailwind CSS v4 + shadcn/ui · Recharts · Supabase(Postgres+Auth) · Google Apps Script · @react-pdf/renderer · Vercel

### 🧗 과정에서의 삽질

#### 결과물 A — Product Design OS

1. **개념 설계가 실사용 첫날 바로 흔들림**: 인박스에 파일이 안 쌓이는 게 이상해 보였는데, 알고 보니 "인박스 = 대기열(처리되면 비워짐)"이라는 설계와 "원본은 남기고 싶다"는 니즈가 충돌. 결국 meetings/(가공 없는 영구 아카이브)를 별도 레이어로 추가하고, 태스크 폴더에는 링크만 남기는 구조로 재설계. 도구는 만들 때가 아니라 실제 데이터를 넣어본 날 가장 많이 배운다.

2. **태스크 단위 원칙을 뒤늦게 발견**: 처음엔 날짜 기반 폴더명으로 설계했는데, 실제로는 "태스크 = GitHub 이슈 1개"가 팀의 작업 단위였다. 이슈가 먼저냐 인풋이 먼저냐 두 경로를 모두 지원하도록 네이밍 규칙(gh-\<번호\>-\<slug\> ↔ 날짜형 + 사후 rename)을 다시 잡음.

3. **두 노트북의 토큰 비대칭 문제**: 지식 추출(코드·Figma 읽기)은 실행 환경에서만 가능. 추출 스킬을 "세션당 3~5개" 단위로 잘게 설계, 스킬이 지식을 통째로 읽지 않고 _MAP.md → 필요한 문서만 여는 progressive disclosure 규칙.

#### 결과물 B — ADMIN 대시보드

1. **"로그인 붙였더니 폼 동기화가 통째로 멈췄다" — RLS 키 지옥 (제일 크게 헤맴)**

   웹 배포를 준비하며 Supabase Auth + RLS를 `로그인 사용자 전용`으로 바꿨더니, 어느 날 구글에서 *"Apps Script 실행 실패"* 메일이 왔습니다. 에러는 `42501: new row violates row-level security policy`.
   - 파고들어 보니 **RLS를 authenticated 전용으로 바꾼 순간, anon 키로 쓰던 폼 동기화가 막힌 것**이 원인. → RLS를 우회하는 `service_role` 키가 필요.
   - 마지막엔 "Disable legacy API keys" 버튼을 누를 뻔했는데, **그걸 눌렀으면 앱과 동기화가 둘 다 죽을** 상황이라 직전에 멈춤.
   - 이 파이프라인은 반드시 **legacy service_role 키**를 써야 했음.

2. **"모바일에서 메일 버튼 누르면 iCloud가 열린다"**

   `mail.google.com/...&view=cm`(작성 화면) URL이 **모바일에서는 무시되고 받은편지함으로 리다이렉트**됨. → `mailto:`로 바꿨더니 이번엔 **iOS 기본 메일 앱(iCloud)**이 열려 발신 계정이 엉뚱해짐. → 최종적으로 **Gmail 앱 딥링크(`googlegmail:///co`)**로 바꿔서 Gmail 작성 화면 + Gmail 계정 발신 확보. (데스크톱은 `authuser`로 계정 고정)

### ✅ 결과물

### 💡 알게 된 인사이트 & 공유하고 싶은 내용

1. **AI 에이전트의 품질은 스킬(동작)이 아니라 맥락(지식 레이어)이 결정한다.** 스킬은 얇게, 지식은 검증 가능한 단위(seed→enriched→verified)로.
2. **워크플로우 도구는 설계할 때가 아니라 실제 데이터를 넣은 첫날 가장 많이 배운다.** 실전 투입을 미루지 말 것.
3. **기능을 붙이는 것보다, 그 기능이 지나가는 '경계'(브라우저↔서버, 데스크톱↔모바일, anon↔service_role 권한)에서 진짜 버그가 터진다.**
