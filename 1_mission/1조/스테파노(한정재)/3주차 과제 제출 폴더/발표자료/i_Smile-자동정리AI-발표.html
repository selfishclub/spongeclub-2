왜 만들었나

손으로 고르고, 배치하고, 각도 맞추고…

교정에서는 내원마다 표준 세트를 촬영해 치료 경과를 평가합니다. 그런데 지금은 낱장 사진을 사람이 직접 골라 배치·정렬하고, 매번 기준이 달라져 전후 비교가 주관적입니다.

🕒 느리다

한 세트에 9~12장을 일일이 찾아 맞는 칸에 넣고 각도를 손으로 보정.

🔀 기준이 흔들린다

배치·크롭 기준이 매번 달라 전후 비교가 들쭉날쭉.

🔒 민감정보

환자 얼굴 사진 — 클라우드 업로드는 부담. 로컬에서만 다뤄야 안전.

💡 다른 분들을 위한 팁

의료 사진 도구는 처음부터 "로컬 전용 · 비파괴"를 원칙으로 설계하세요. 원본은 바이트 그대로 보존하고, 보정은 항상 원본에서 다시 계산해 별도 저장 — 그래야 언제든 되돌리고, 민감정보가 밖으로 나갈 일이 없습니다.

AI 자동 배열 원리

"분류" + "한 칸에 하나씩" 규칙

두 단계로 동작합니다. 먼저 작은 CNN이 각 사진의 종류를 추측하고, 그다음 한 종류당 한 장이라는 규칙으로 전체를 가장 잘 맞는 조합으로 배치합니다.

낱장 사진 뭉치순서 제각각 · 이름 제각각
→
CNN 분류12종 종류 + 확신도
→
한 칸에 하나씩헝가리안 최적 매칭
→
완성 보드확인 후 저장

왜 "한 칸에 하나씩"이 중요한가

사진을 한 장씩만 보면 두 장이 같은 칸을 노려 겹칠 수 있어요. "한 세트엔 종류마다 딱 한 장"이라는 당연한 규칙(헝가리안 매칭)으로 서로 안 겹치게 전역 최적 배치 → 보드가 깔끔하게 완성됩니다. (데이터가 적던 초기엔 정확도도 +5%p 올려줬어요.)

모든 게 기기 안에서

모델은 ONNX로 앱에 내장, onnxruntime(CPU) 로 추론. 인터넷 연결이 전혀 필요 없습니다. Windows 8.1처럼 옛 PC는 자동 기능 없이 수동으로 그대로 쓸 수 있게 분기했습니다.

💡 다른 분들을 위한 팁

순수 딥러닝에만 기대지 말고 문제의 "구조·규칙"을 제약으로 넣으세요. 여기선 "세트당 종류별 1장"(헝가리안 1:1 매칭)으로 겹침을 막고 전역 최적 배치를 얻어요. 도메인 지식이 곧 공짜 데이터 — 데이터가 적을수록 특히 큰 힘이 됩니다.

정확도 · 정직하게

라벨을 늘리니 90%까지

사진 1장 단위89.9%
보드 단위 (한 칸에 하나씩)87.2%
X-ray 종류 (파노·세팔로·Wrist)~100%

라벨을 750장 → 2,800여 장으로 늘리니 68% → 89.9%로 상승. 9컷 12종 + X-ray 5종(16종)까지 인식. X-ray는 생김새가 확연히 달라 거의 100%. 남은 약점: 좌/우 구치·전하방45°(라벨 더).

완벽하지 않아도 쓸모 있게 — 확인형 UI

자동은 제안일 뿐. 배치된 칸을 확신도 색으로 표시해 "봐야 할 곳"을 바로 알려줍니다. 9칸 중 6~7칸이 미리 맞으니 손이 크게 줍니다.

확실 — 그냥 넘김 보통 — 슬쩍 확인 낮음 — 꼭 확인
💡 다른 분들을 위한 팁

정확도를 부풀리지 말고 "정직한 숫자 + 안전장치"로 설계하세요. 90%짜리라도 "확신 낮은 칸만 빨갛게" 표시하면 더 믿음직해집니다. 중요한 건 사람이 항상 덮어쓸 수 있다는 것 — 오분류가 사고로 이어지지 않게.

개발 중 만난 벽 · 실전 교훈

진짜 시간을 잡아먹은 것들

기능보다 이런 "사소해 보이는" 문제들이 발표를 위협했습니다. 남들도 똑같이 겪을 것들이라 강조합니다.

1AI 스레드가 화면을 굶긴다

자동 배열을 몇 번 하니 앱 전체가 멈춘 듯 느려짐. 원인은 onnxruntime CPU 스레드가 추론이 끝난 뒤에도 모든 코어를 계속 점유(busy-spin). 스핀 끄고 스레드 수 제한 → 해결.

2세로로 긴 세팔로가 눌린다

비교 화면이 첫 사진 비율로 전부 리사이즈해 세팔로가 찌그러짐. 각 사진이 자기 비율을 지키도록(레터박스) 수정.

3창이 화면보다 커서 아래가 잘림

창을 900px 고정으로 열어 노트북 화면 밖으로 하단이 나감. 화면 크기에 맞춰 열고, 좌측 목록은 스크롤로.

4비교 화면에 사진이 안 보임

파일명이 날짜인 것만 보여주던 필터 때문에 DSC_0001 류가 전부 숨겨짐. 이름과 무관하게 항상 보이도록 + 손상 파일에도 안 죽게 방어.

🧨 가장 값진 교훈 — "안 바뀌었어요!"의 정체

고쳐서 새로 빌드해도 사용자는 계속 옛 버전을 보고 있었습니다. 원인: 압축을 같은 폴더에 반복 해제해 macOS가 앱 2.app, 앱 3.app… 복사본을 7개 만들었고, 매번 맨 처음 것을 열고 있었죠. 교훈 ▶ 버전 번호를 화면에 노출하고(제목표시줄), 배포 파일명에도 버전을 박아 "지금 이게 최신인지"를 1초 만에 확인할 수 있게 하세요.

핵심 — 라벨링 노가다 없애기

"처음부터 라벨 달기"를 그만두다

인식률을 올리려면 데이터가 필요한데, 800장을 한 장씩 손으로 라벨링하는 건 무리. 그래서 사람의 역할을 "라벨링"에서 "검수(틀린 것만 고치기)"로 바꿨습니다.

🥵 예전 — 노가다

  • 한 장 뜰 때마다 1~9 판단
  • 800장 = 800번의 결정
  • 지치고, 뒤로 갈수록 실수
▶

😌 지금 — 검수

  • 모델이 미리 다 분류해 추천을 띄움
  • 맞으면 Enter, 틀린 것만 숫자키
  • 약 70%는 Enter 한 번 → 손품 70%↓

①빠른 검수 도구

모델이 확신 높은 순으로 예측을 미리 선택해 보여줌. Enter 수락 · 숫자 수정 · → 보류. 밀린 백로그를 훨씬 적은 클릭으로.

②앱 쓰면서 자동 축적

앱에서 자동 배열 → 고치고 → 저장만 하면 그 9칸 배치가 그대로 학습 라벨로 수확됩니다. 별도 라벨링 시간 0 — 진료가 곧 라벨링.

③헷갈리는 것만 먼저

모델이 확신 없는 것만 골라 물어보게(액티브 러닝). 같은 노력으로 정확도가 가장 빨리 오릅니다.

④자기학습 자동 보강

모델이 아주 확신하는 사진은 사람 없이 라벨 추가. 잘 틀리는 종류는 제외하고, 별도 파일로 저장해 안전하게.

쓸수록 라벨이 쌓이고, 재학습으로 정확도가 올라 검수 때 Enter 비율이 높아져 손이 점점 덜 갑니다.

💡 다른 분들을 위한 팁

데이터 라벨링이 벽이라면, 이미 있는 (약한) 모델을 라벨링에 되먹이세요. "모델이 먼저 찍고 → 사람은 틀린 것만" + "제품을 쓰는 행위 자체가 라벨을 만들게" 설계하면, 노가다가 자동화되고 데이터가 저절로 굴러갑니다.

무엇으로 만들었나

가볍고, 로컬에서 도는 스택

🖥️ 데스크톱 앱

Python + PySide(Qt). mac(.app)·Windows(.exe) 한 소스로. 옛 Win8.1은 Qt5로 분기.

🧠 자동 분류

MobileNetV3 전이학습 → ONNX 내장, onnxruntime CPU 추론. 헝가리안 매칭(순수 파이썬).

🖼️ 이미지

Pillow(HEIC 포함), 비파괴 보정, 내보낼 때만 선택적 비식별화·EXIF 제거.

💡 팁

한글 파일명·맥 심볼릭 링크는 Windows 압축 해제를 깨뜨립니다. 배포 zip은 순수 ASCII 이름 + 심볼릭 링크 0으로. 플랫폼별 함정(옛 OS엔 ML 라이브러리 미지원 등)을 미리 분기해 두면 배포 사고를 크게 줄입니다.
