> 미션을 진행하며 **기획 → 구현 → 삽질 → 결과물 → 인사이트** 를 상세히 기록해주세요.

## 🎯 미션 1. 내 OS 만들기
> **[ 내 삶을 돕는 OS ]** 또는 **[ 콘텐츠 OS ]** 중 하나를 선택해 완성해주세요.

**✅ 선택:** 흔적 OS  _(유형: 내 삶을 돕는 OS)_

**흔적 OS — 내가 시스템을 돌보는 게 아니라, 시스템이 내 흔적을 돌보는 OS**

### 📐 기획
> 무엇을, 왜, 어떻게 만들지

#### 한 문장
> **"흘러가는 일상이 저절로 쌓이는 OS."**
> 내가 시스템을 돌보는 게 아니라, **시스템이 내 흔적을 돌봐주는** 구조.

#### 무엇을 (What)
지나가는 생각·기록을 **가볍게 던지기만 하면**, 시스템이 알아서 줍고·분류하고·자산과 콘텐츠의 재료로 쌓아주는 개인 OS. 나는 하루 안에서 **본업 · 글쓰기/콘텐츠 발행 · 사이드 프로젝트 · 엄마**라는 4개의 역할을 오가며 산다. 이 역할들을 "더 잘 관리하려고 에너지를 쓰는" 시스템이 아니라, **지나가는 것을 대신 주워 담아주는** 시스템을 만든다.

#### 왜 (Why) — 기대하는 효능
역할을 전환할 때마다 머릿속을 **재부팅하는 비용**을 없애는 것. 일 → 가정 → 사이드로 넘어갈 때마다 그 역할의 맥락을 처음부터 다시 불러오는 데 생각보다 많은 에너지가 샌다.

- 이 **전환 비용이 사라지면** → 여러 역할의 병렬 운영이 애쓰지 않아도 따라오는 결과가 된다.
- 하루가 **증발하지 않고 자산으로 남는 감각**을 얻는다.

#### 어떻게 (How) — 2기 수료 전까지의 목표 형태
**캡처 → 분류 → 복원**, 이 세 단계가 실제로 도는 **최소 루프**를 만든다.

| 단계 | 무엇을 | 기준 |
|------|--------|------|
| **① 캡처(Capture)** | 떠오르는 생각·글감·할 일을 **텔레그램에 한 줄** 던진다 | **30초를 넘지 않을 것** |
| **② 분류(Classify)** | Claude가 던진 한 줄을 **역할별로(본업/콘텐츠/사이드/엄마) 분류해 쌓는다** | 내가 관리하지 않아도 유지될 것 |
| **③ 복원(Restore)** | 어떤 역할로 **전환하는 순간**, 그 역할의 현재 상태를 **한 화면으로 복원**해준다 | 맥락 재부팅 없이 바로 이어가기 |

**핵심 제약(설계 원칙):**
- 캡처는 **30초 이내** — 무거우면 안 쓴다.
- **무관리 유지** — 내가 정리·관리하지 않아도 시스템이 스스로 쌓인다.
- **일상에 던지는 것 자체가 곧 구축** — 하루 1시간을 시스템에 "쏟는" 게 아니라, 일상을 지나가며 던지는 행위가 OS를 자라게 한다.

#### 구현 로드맵 (작게 → 크게)
| 단계 | 범위 | 이번 2주차 |
|------|------|:---:|
| **1단계** | 텔레그램 → Claude 캡처 파이프라인 (한 줄 던지면 저장) | 🎯 착수 |
| **2단계** | 역할별 자동 분류 (본업/콘텐츠/사이드/엄마) | 다음 |
| **3단계** | 역할 전환 시 '현재 상태 한 화면' 복원 뷰 | 다음 |

### ⚙️ 구현
> 실제로 만든 것 (링크·스크린샷 — 이미지는 `이미지첨부/` 폴더에)

**만든 것: 흔적 OS의 "두뇌" + 역할별 저장 구조** (폴더: [`흔적-os/`](./흔적-os/))

캡처→분류→복원 세 단계 중, 나머지 단계의 재료가 되는 **분류·복원 두뇌**를 먼저 구현했다. 텔레그램(입구)은 봇·서버 세팅이 필요해 2단계로 미루고, 지금 클로드 코드 안에서 바로 작동·테스트되는 두뇌부터 만들었다. 이 두뇌는 나중에 텔레그램이 붙어도 그대로 재사용된다.

- **`skill/SKILL.md`** — 흔적 OS의 두뇌(클로드 코드 스킬). 두 가지 모드:
  - **담기(분류):** 흘려보낸 한 줄 → `역할(본업/콘텐츠/사이드/엄마/미분류)` + `종류(할일/생각/글감/자료)` 판단 → 해당 `roles/{역할}.md`에 한 줄 append. 애매하면 버리지 않고 `미분류`로 보관.
  - **꺼내기(복원):** 역할 이름을 부르면 → 그 역할의 열린 루프·최근 글감을 **딱 한 화면**으로 압축 브리핑.
- **`inbox.md`** — 던지는 곳(캡처). 30초 이내 한 줄. (2단계에서 텔레그램 유입구로 연결)
- **`roles/*.md`** — 역할별로 쌓이는 저장소 (본업·콘텐츠·사이드·엄마·미분류).
- **`README.md`** — 루프 개요·사용법·로드맵.

**동작 테스트 (뒤섞인 7줄 던지기 → 자동 분류):**

| 던진 한 줄(원문) | → 역할 | 종류 |
|---|---|---|
| OpenAI 새 에이전트 플랫폼 임원 보고 다음주 화요일까지 초안 | 본업 | 할일 |
| 규제산업 1등 포지셔닝, 블로그 글로 풀면 좋겠다 | 콘텐츠 | 글감 |
| 사이드 PoC 사용자 인터뷰 5명, 질문지 아직 없음 | 사이드 | 할일 |
| 금요일 아이 어린이집 현장학습 준비물 도시락+모자 | 엄마 | 할일 |
| '역할 전환 비용' 개념, 뉴스레터 소재로 | 콘텐츠 | 글감 |
| 경쟁사 벤치마크 자료 https://… | 본업 | 자료 |
| 아침 재활 운동 15분 루틴 지키기 | 미분류 | 할일(건강) |

**복원 출력 예시 (역할 전환 시 '한 화면'):**
```
🟢 [본업] 복원 — 2026-07-09
▶ 지금 열린 루프: OpenAI 임원 보고 초안 — 화요일까지
▶ 최근 글감: (없음)
▶ 한 줄 브리핑: "지금 본업에서 중요한 건 화요일 임원 보고 초안이다."
```

**2단계 — 텔레그램 자동 캡처 파이프 (🟢 실배포 완료·실동작 확인):** 폴더 [`흔적-os/telegram-capture/`](./흔적-os/telegram-capture/)
- 텔레그램에 한 줄 보내면 → **Vercel 서버리스 함수**가 받아 → **내 개인 비공개 저장소**(`imprint-os-data`)의 `inbox.md`에 append → "📥 담았어요 ✅" 답장. (진짜 "30초 캡처")
- **실제로 켜서 확인:** 봇 `@imprint_os_bot` 생성 → Vercel 배포(`imprint-os-capture.vercel.app`) → 텔레그램 webhook 연결 → **실제 메시지를 보내니 봇이 "담았어요 ✅" 답장 + 개인 저장소 `inbox.md`에 그 줄이 자동으로 커밋됨을 확인.** 🎉
- **설계 판단:** 캡처 데이터는 반 전체가 보는 공용 저장소가 아니라 **개인 비공개 저장소**로 분리(프라이버시). Vercel은 '분류 안 하는 튼튼한 파이프'만 맡고, 분류·복원은 두뇌(스킬) 한 곳에만. 비밀키(봇·깃허브 토큰)는 코드가 아니라 **Vercel 환경변수**에만.
- **검증 2단계:** ① 배포 전 가짜 텔레그램 요청으로 파이프 전 구간 모킹 테스트 `✅ PASS` → ② 배포 후 실제 텔레그램 메시지로 엔드투엔드 `✅ 확인`.
- 설치 재현 안내: [`telegram-capture/SETUP.md`](./흔적-os/telegram-capture/SETUP.md) (봇·저장소·토큰·Vercel·webhook 30분 따라하기)

### 🧗 과정에서의 삽질
> 막혔던 지점, 시도한 방법, 어떻게 풀었는지 솔직하게

- **처음 막힌 지점:** "일상이 저절로 쌓이는 OS"라는 그림이 커서, 어디부터 손대야 할지 막막했다.
- **어떻게 좁혔나:** 캡처(텔레그램)를 먼저 만들려다, **입구(텔레그램 봇·서버)는 계정·배포 세팅이 필요해 오늘 안에 '작동'까지 못 간다**는 걸 깨달았다. 그래서 순서를 뒤집어, **계정 없이도 지금 돌아가고 테스트되는 '두뇌(분류·복원)'** 부터 만들었다. 두뇌는 텔레그램이 붙어도 그대로 재사용되므로 헛일이 아니다.
- **저장 형식 고민:** 처음엔 역할별로 그냥 쌓기만 했는데, 그러면 '복원'이 장황해졌다. 각 역할 파일을 `열린 루프 / 생각·글감 / 자료` 3칸으로 나누니, 복원할 때 "먼저 처리할 것(열린 루프)"만 위로 뽑아 **한 화면**으로 압축할 수 있었다.
- **애매한 줄 처리:** "재활 운동 루틴"처럼 4개 역할에 안 걸리는 게 나왔다. 버리지도, 억지로 끼우지도 않고 **`미분류`로 보관**하는 규칙을 두뇌에 넣어 해결.

**2단계 실배포하며 겪은 삽질 (실제 세션 기록):**
- **저장소를 2개로 나눔:** 처음엔 캡처를 과제 저장소에 넣으려다, 그러면 내 생각·아이 일정이 반 전체에 공개된다는 걸 깨닫고 → **코드용(공개) / 데이터용(비공개)** 저장소를 분리. 코드엔 비밀이 없고, 비밀키는 Vercel 환경변수로만.
- **Vercel 새 UI 헤맴:** "Environment Variables" 메뉴가 안 보여 한참 찾음 → 새 UI에선 **`Environments`** 안에 있었다.
- **403의 정체:** 배포 후 봇 주소가 처음엔 안 열리는 듯 보였는데, 알고 보니 접속하는 쪽 네트워크 문제였고 **브라우저로 직접 열어보니 "캡처봇 살아있음" 정상 응답**. → "안 되는 것처럼 보인다≠안 된다", 다른 경로로 확인하는 습관.
- **webhook 연결의 함정:** 텔레그램 연결 주소에서 `bot` 글자 뒤에 토큰을 붙여야 한다는 규칙(`bot<토큰>`)을 놓치기 쉬웠다. `{"ok":true}` 응답으로 확인.
- **클로드 코드 ↔ 비공개 저장소의 벽:** 두뇌를 자동으로 돌리려 `imprint-os-data`를 클로드 코드 세션에 붙이려 했는데, **공개 저장소는 보이지만 비공개 저장소는 접근 권한이 없어** 목록에 안 떴다. 그래서 이번엔 두뇌의 분류 결과를 **직접 커밋**해 실제 `roles/` 파일을 만들었다(결과는 자동과 동일). 완전 자동("인박스 정리해줘" 한마디)은 비공개 저장소 연결을 1회 세팅하면 되는 다음 과제로 남겼다.

### ✅ 결과물
> 완성한 것 / 작동 화면

**흔적 OS 1단계 완성 — 캡처→분류→복원 루프의 '두뇌'가 실제로 돈다.**
- 폴더: [`흔적-os/`](./흔적-os/) (`skill/SKILL.md` 두뇌 · `roles/*.md` 저장소 · `inbox.md` 캡처 입구 · `README.md`)
- 검증: 뒤섞인 7줄 → 역할별 자동 분류 완료 → 역할 하나만 부르면 그 맥락만 한 화면으로 복원됨을 확인. ✅
- 확인 지점: 각 역할 파일([`roles/본업.md`](./흔적-os/roles/본업.md) 등)에 분류된 결과가 실제로 쌓여 있음.

**🟢 전체 루프 가동 확인:** 두뇌 스킬(`흔적-분류-복원`)을 개인 데이터 저장소 `imprint-os-data/.claude/skills/` 에 등록 → 이제 **캡처(텔레그램·자동) → 분류("인박스 정리해줘") → 복원("본업 복원")** 세 조각이 모두 실제로 연결돼 돈다.

**🟢 첫 실사용 가동 (실데이터로 확인):** 텔레그램으로 실제 3줄을 던지고(뉴스레터 글감 / AX 과제 점검 회의 자료 / 흔적 OS 블로그 후기), 두뇌로 역할을 분류했다. 그 결과 개인 저장소 `imprint-os-data` 에 **`roles/콘텐츠.md`(글감 1 + 할일 1) · `roles/본업.md`(할일 1)** 가 실제로 생성되고 `inbox.md` 가 비워짐을 눈으로 확인. → 예상(미리보기)과 실제 결과가 일치. 흔적 OS가 '기획'이 아니라 '실제로 도는 것'이 됨.

**로드맵 대비 위치:** ✅ 1단계(두뇌+저장) 완료 → ✅ 2단계(텔레그램 캡처 파이프 **실배포·실동작 완료**) → ⬜ 3단계(캡처 시 실시간 자동 분류 / 역할 전환 시 복원 자동 푸시)

### 💡 알게 된 인사이트 & 공유하고 싶은 내용
> 하면서 깨달은 것, 크루들과 나누고 싶은 것

- **"관리하는 시스템"과 "돌봐주는 시스템"은 다르다.** 대부분의 생산성 도구는 내가 그 도구를 관리해야 유지된다. 내가 진짜 원한 건 관리 대상이 하나 더 늘어나는 게 아니라, **내 흔적을 대신 주워 담아주는** 쪽이었다.
- **가장 비싼 비용은 '역할 전환 시 맥락 재부팅'이었다.** 일이 많아서가 아니라, 역할을 오갈 때마다 머리를 다시 켜는 데 에너지가 샌다는 걸 언어로 잡아낸 게 이번 기획의 핵심.

- **입구(캡처)보다 두뇌(분류·복원)를 먼저 만든 게 신의 한 수.** 눈에 보이는 '텔레그램 봇'을 먼저 만들고 싶은 유혹이 컸지만, 그건 계정·배포에 막혀 오늘 '작동'을 못 봤을 것이다. 계정 없이도 지금 돌아가는 두뇌부터 만드니, **첫 세션 안에 실제로 작동하는 결과**를 손에 쥐었다. → 만드는 순서는 "화려한 입구"가 아니라 "혼자서도 도는 코어"부터.
- **복원은 '많이 보여주기'가 아니라 '한 화면으로 자르기'였다.** 다 보여주면 그게 또 다른 재부팅이다. 열린 루프 top + 최근 글감 3개로 잘라야 진짜 '복원'이 된다.
- **'된다'와 '편하게 된다'는 다른 단계다.** 두뇌 로직은 첫날 완성됐지만, 그걸 매일 한마디로 굴리는 '편의(자동 연결)'는 또 다른 세팅이 필요했다. 오늘은 손으로 직접 돌려 결과를 확인했고 — 그것만으로도 "기획이 실제로 도는 것"이 되는 문턱을 넘었다. **작동하는 것 → 손에 익는 것** 사이의 거리를, 몸으로 배운 하루.
