# G엔터 나라장터 입찰 자동화 시스템 (genent-nara-bid)

> v2 (2026-05-17 재작성). v1은 G엔터 측 자료 수신 전 추정 설계였고, 본 v2는 G엔터가 제공한 협업 SOP 폴더(`genent_bid_cowork`)를 reference로 통합·정합화한 최종 설계다.

## 목적

협업사 **주식회사 G엔터(G ENT, 대표이사)** 의 명의로 응찰 가능한 나라장터 입찰공고를 매일 자동 탐색하고, G엔터가 미리 정리한 협업 SOP에 따라 **응찰 가능 여부 자동 판정 → RFP 분석 문서 자동 작성 → 제안서 초안 자동 생성** 까지 단계별로 진행한다.

우리 회사(대표)와 G엔터(컨택: 담당자)가 공유 Notion DB에서 협업하며, 최종 검수·제출은 사람이 한다.

## 기존 시스템과의 구분

- **`gov-subsidy-scout`** (2026-04-13 spec): 기업마당 API → 우리 회사 법인 자체 신청용 지원사업. 별개.
- **본 spec**: 나라장터(g2b) API → G엔터 입찰 제안서. 협업사 운영 자동화.

## v1 → v2 변경 요지

v1은 "공고 첨부 docx 양식 빈칸 채움" 패러다임이었으나, G엔터 측이 이미 다음을 정리해서 제공:

- **회사 자산 폴더** (`01_company_assets/`): 회사 프로필·실적·인력·크리에이터DB
- **재사용 템플릿** (`05_templates/`): 마스터 제안서(A4 100p 9챕터) + 섹션 템플릿 8개 + 체크리스트
- **입찰 지식 베이스** (`06_playbook/`): 나라장터 가이드, 공공 문체 가이드, 평가 패턴, 용어집, 제출 체크리스트
- **명확한 응찰 룰** (`00_CLAUDE.md`): 단독 정공법, 5천만 이하 우선, 자동 제외 조건 5종

v2는 위 자료를 **그대로 reference로 활용**한다. 자동화 시스템은 cowork 폴더 구조에 부속되며, 제안서 생성은 챕터별 LLM 생성 패러다임으로 전환.

## 핵심 사실 (cowork 자료 기반)

### 회사 프로필
- 회사명: 주식회사 G엔터 (G ENT.)
- 대표자: 대표이사
- 설립: 2024.09.24, 사업자등록번호 [정보 비공개]
- 본사: [주소 비공개]
- 자본금: n만원, 정직원 n명, 매출 2025년 n원 (전년대비 n%)
- 업종: 광고대행업·소프트웨어 개발·경영 컨설팅
- 핵심 자산: 크리에이터 DB n명+, 카카오톡 공식채널 친구 n명, 검증 인플루언서 풀 n명

### 입찰 전략 방침 (2026-04 확정)
- **단독 정공법**: 공동수급 거절, 단독 입찰만 (예외: G엔터가 주계약자가 되는 경우)
- **타겟**: 추정가격 5천만원 이하 + 단일건 실적 요건 없음 + 재무 3개년 미요구 사업
- **우선 발주처**: 지자체 산하기관·재단법인, 교육청·공공기관 소규모, 중앙부처 산하 연구기관
- **특별 우선 타겟**: 의료관광·외국인 관광객 유치 (샤오홍슈 왕홍 실적), 뷰티·여성·가족·육아 정책, AI·디지털 전환
- **로드맵**: 1단계(2026.하~2027.상) 5천만 이하 3~5건 낙찰 → 2단계 1억대 → 3단계 1.5억+

### 포지셔닝 3종
- **A. SNS 홍보대행 전문**: n명 크리에이터 네트워크, 다양한 업종 수행 — 보수적 발주처용
- **B. AI 기반 인플루언서 플랫폼**: AI 알고리즘 자체개발, 자동화 시스템 — 혁신 발주처용
- **H. 중화권·해외 마케팅 (하이브리드)**: 샤오홍슈 왕홍 직접섭외 실적, 광저우 창롱 팸투어 — 의료관광·외국인 관광용

```

    ↓ (매일 09:00)
[poll_nara.py]
    ├─ 신규 공고 폴링

[dispatcher (매 5분) → rfp_analyzer.py]

    ├─ 03_proposals/{사업폴더}/ 생성
    ├─ Sonnet으로 마스터 템플릿 9챕터 + 섹션 템플릿 + 회사자산 결합
```

## 자동화 컴포넌트

### 1. `scripts/poll_nara.py` — 매일 09:00

**입력**: 조달청 OpenAPI `용역 입찰공고` 어제 09:00 ~ 오늘 09:00 신규 등록건

**파이프라인**:

1. **신규 공고 수집** — 분류 "용역 > 광고·홍보, 정보화, 콘텐츠 제작" 우선
2. **중복 제거** — `data/seen_bids.json` 조회
3. **1차 카테고리 키워드 필터** (저비용, 규칙 기반):
   - 콘텐츠·SNS: `SNS`, `누리소통망`, `홍보`, `홍보대행`, `콘텐츠 제작`, `숏폼`, `유튜브`, `인스타그램`, `온라인 홍보`, `채널 운영`, `릴스`, `소셜미디어`
   - 인플루언서: `인플루언서`, `크리에이터`, `체험단`, `KOL`, `왕홍`
   - 글로벌: `해외 홍보`, `K-콘텐츠`, `샤오홍슈`, `小紅書`, `외국인 관광`, `중화권`
   - 특별 우선 가점: `의료관광`, `뷰티`, `여성`, `가족`, `육아`, `AI`, `디지털 전환`
4. **응찰 가능 자동 판정 (필수 룰)** — 다음 중 하나라도 해당 시 `상태 = 응찰불가`
   - 추정가격 ≥ 3억원
   - 본문에 "단일건 5천만원 이상 실적" 또는 동등 패턴 검출
   - "재무제표 3개년" 또는 "최근 3개년 결산" 요구 검출
   - "공동수급 의무" / "컨소시엄 의무" 검출
   - "직접생산확인증명서" 요구 (단, 동영상제작서비스 8213160301는 향후 발급 시 재검토 메모)
5. **2차 LLM 매칭 + 포지셔닝 추천** (Haiku, ~700 토큰/건):
   - 입력: 공고명·요지·예산·기관 + 회사 프로필 200자 요약 + 포지셔닝 A/B/H 정의
   - 출력 JSON: `{score: 0~100, position: "A|B|A+H|B+H", category: [], rationale: "한 문장", flags: ["응찰가능", "특별우선타겟" 등]}`
6. **Notion 적재**: 상태별 자동 분류
   - `응찰불가` → 필터로 숨김 (히스토리만 보존)
   - 점수 70+ AND 5천만 이하 AND 특별우선타겟 → `검토중(강력권장)`
   - 점수 70+ AND 5천만 이하 → `검토중(권장)`
   - 점수 30~70 → `검토중`
   - 점수 < 30 → `참고` (자동 숨김)
7. **`data/seen_bids.json` 업데이트**

**비용**: 일 1차 필터 통과 5~10건 × Haiku ~$0.002 = 일 $0.02 미만.

### 2. `scripts/rfp_analyzer.py` — RFP 분석 (사용자 클릭 시)

**트리거**: Notion `RFP 분석` 체크박스 ON

**파이프라인**:

1. **상태 잠금**: `RFP 분석 = FALSE`, `상태 = 분석중`
2. **사업 폴더 생성**: `02_rfp_archive/{발주처}_{사업명}_{연도}/`
3. **첨부파일 다운로드**: 조달청 API 첨부 URL → `rfp_original.{ext}` 저장
4. **포맷 변환**:
   - HWPX (XML) → `python-hwpx` 직접 처리
   - HWP → LibreOffice headless `soffice --convert-to docx`
   - PDF → `pdfplumber` 텍스트 추출
   - 변환 실패 → 텍스트 추출만 + `상태 = 변환실패` 알림
5. **6 STEP 자동 분석** (Sonnet, ~3000 토큰):
   - Context 주입: `00_CLAUDE.md` (룰) + `06_playbook/*` (평가패턴·문체·용어) + 변환된 RFP 텍스트
   - 출력 1: `rfp_analysis.md` (template의 6 STEP 모두 채움)
   - 출력 2: `evaluation_criteria.md` (STEP 3만 별도 추출)
6. **Notion 업데이트**:
   - `응찰권장등급`: STEP 6의 종합 판정
   - `예상승률`: STEP 6
   - `포지셔닝확정`: STEP 5
   - `RFP 분석 링크`: 사업 폴더 경로
   - `상태`: `분석완료`
7. **iMessage 알림** (선택): "응찰 강력 권장" 등급만 사용자에게 통보

**비용**: 1건당 Sonnet ~$0.30~$0.60.

### 3. `scripts/proposal_builder.py` — 제안서 초안 생성

**트리거**: Notion `제안서 초안` 체크박스 ON (단, `분석완료` 상태에서만 허용)

**파이프라인**:

1. **상태 잠금**: `제안서 초안 = FALSE`, `상태 = 생성중`
2. **결과 폴더 생성**: `03_proposals/{사업폴더}/`
3. **9 챕터 자동 생성** (마스터 템플릿 기준):
   - 각 챕터별 Sonnet 호출 (병렬 가능)
   - Context: 해당 챕터 섹션 템플릿 + 회사 자산 관련 부분 + RFP 분석 결과 + 평가기준 가중치
   - 포지셔닝(A/B/H)에 따라 회사소개·차별점 텍스트 다르게 사용
   - 공공 문체 가이드 (`writing_style_guide.md`) 시스템 프롬프트로 항상 포함
4. **챕터 결합**: python-docx로 9챕터 → 단일 docx 조립
   - 표지·제출문 자동 생성
   - 페이지 일련번호 자동 삽입
   - 표·다이어그램 placeholder 표시 (사람이 채울 부분 명시)
5. **저장**: `03_proposals/{사업폴더}/draft_v1.docx`
6. **GDrive 백업**: `[우리 회사] G엔터/제안서 초안/` (메모리 `feedback_artifact_storage_policy` 준수)
7. **Notion 업데이트**:
   - 파일 첨부 + GDrive 링크
   - `LLM 비용` 누적
   - `상태`: `초안완료`

**비용**: 1건당 Sonnet ~$1.5~$3 (챕터 9개 × 평균 분량).

### 4. `scripts/notion_sync.py` — 공통 라이브러리

## 응찰 가능 자동 판정 룰 (`filter_rules.py`)

`00_CLAUDE.md` "응찰 불가 사업 유형"을 코드화. 각 룰은 정규식 + 키워드 기반.

| 룰 | 검출 패턴 | 동작 |
|---|---|---|
| R1. 단일건 실적 5천 이상 | `단일\s*건.*5천만원`, `최근\s*\d+년.*5천만원\s*이상`, `금\s*5천만원\s*이상.*실적` | 응찰불가 |
| R2. 재무 3개년 | `최근\s*3개년\s*재무제표`, `3개\s*사업연도.*재무` | 응찰불가 |
| R3. 단일 추정 3억 이상 | `추정가격`/`사업예산` 숫자 추출 ≥ 300,000,000 | 응찰불가 |
| R4. 공동수급 의무 | `공동수급.*의무`, `컨소시엄.*필수`, `공동이행.*의무` | 응찰불가 |
| R5. 특수 자격 (직접생산) | `직접생산확인증명서.*필수`, `소프트웨어사업자.*신고` | 응찰불가 (재검토 메모) |
| R6 가점. 5천만 이하 | 추정가격 ≤ 50,000,000 | 가점 +20 |
| R7 가점. 소기업 제한 | `소기업·소상공인.*제한경쟁`, `소기업\s*제한` | 가점 +15 |
| R8 가점. 특별 우선 타겟 | `의료관광`, `외국인\s*관광`, `뷰티`, `K-?컬처`, `AI.*홍보` | 가점 +10 |

LLM 매칭 점수 + 가점 합산 → Notion 최종 점수.

## 빌드 단계 (Phase)

- **Phase 0** (사람 액션): 나라장터 OpenAPI 키 발급 (우리 회사 명의), `.env` 작성, Notion DB 생성·권한 부여
- **Phase 1**: `poll_nara.py` + `filter_rules.py` + Notion 적재 (RFP 분석·제안서 생성 제외)
- **Phase 2**: `rfp_analyzer.py` + HWPX 처리 (구버전 HWP는 Phase 3)
- **Phase 3**: HWP 구포맷 LibreOffice 변환 + 변환 실패 Plan B
- **Phase 4**: `proposal_builder.py` 9챕터 생성 + docx 조립
- **Phase 5**: launchd 등록 + 30일 운영 검증

## 리스크

| 리스크 | 영향 | 대응 |
|---|---|---|
| HWP 변환 실패 (~15~30%) | RFP 분석 정확도 저하 | Plan B 텍스트 추출, 사람 수동 정리 |
| 자동 제안서 docx 그대로 제출 가능률 30~50% | 사람 다듬기 필요 | 시스템 목표: "0 → 70% 초안 자동화" |
| LLM이 회사 자산을 잘못 인용 (없는 실적 환각) | 신뢰 손상 | 시스템 프롬프트에 "제공된 자산 외 사실 만들지 마라" 명시, 사용자 검수 필수 |
| 응찰 가능 판정 오류 (regex 한계) | 응찰 기회 누락 또는 시간 낭비 | LLM 보조 검증, Notion에서 사람 최종 판단 |
| Notion API rate limit | dispatcher 실패 | 지수 백오프, 5분 cron이라 영향 미미 |
| LLM 토큰 폭발 (제안서 1건 $3+) | 비용 증가 | 분석 단계 강제 (분석완료 없이 초안 생성 불가), Notion `LLM 비용` 누적 추적 |
| G엔터 자산 업데이트 지연 | 오래된 정보로 제안서 작성 | 분기 1회 G엔터 측에 자산 갱신 요청, 자동 알림 |

## 검증 기준

- **Phase 1**: 7일 운영 후 Notion 일 평균 매칭 적재 ≥ 3건, `응찰불가` 자동 판정 정확도 ≥ 90% (사용자 샘플 검수)
- **Phase 2**: HWPX 파일 분석 5건 성공, `rfp_analysis.md` 6 STEP 모두 자동 채움 (placeholder 미해결 < 10%)
- **Phase 3**: HWP 구포맷 변환 성공률 ≥ 70%
- **Phase 4**: 제안서 초안 5건 자동 생성, G엔터 검토 "수정 후 제출 가능" 판정 ≥ 3건
- **Phase 5**: 30일 무중단 가동, 자동화 실패율 < 10%, 응찰 검토 진입 ≥ 5건

## G엔터 측 선결 과제 (시스템 외)

4. 직접생산확인증명서 (동영상제작서비스 8213160301) 발급 검토
5. 나라장터 입찰참가자격 등록 (광고대행업 9902 + 기타자유업 9999 이중 등록)
