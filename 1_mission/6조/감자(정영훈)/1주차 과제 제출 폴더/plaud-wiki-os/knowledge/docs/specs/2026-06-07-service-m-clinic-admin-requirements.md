- 작성일: 2026-06-07
- 대상: 개발

## 1. 개요

### 1.1 목적

파트너 병원이 자기 예약·환자를 직접 관리하고, 공개 클리닉 페이지를 직접 수정하는 어드민을 제공한다.

> **예약 유입 구조 (전제 — 코드 확인 완료 2026-06-07)**: 모든 예약은 자체 booking 폼(`Booking` 테이블) 한 곳으로 들어온다. 구글 지도 "Book Online"은 Google Actions Center action feed를 통해 이 자체 폼으로 redirect되는 것이며, 별도 데이터 소스가 아니다. 따라서 어드민의 예약 데이터 소스는 `Booking` 단일이다.

### 1.2 사용자

- **파트너 병원** (단일 사용자 그룹). 각 병원은 **자기 데이터만** 접근. 병원 간 완전 격리.
- 병원 내 역할 분리(원장/실장 등)는 **이번 범위 아님** — 병원당 단일 계정 권한.

### 1.3 기존 시스템과의 구분 (중요)

- 기존 `apps/web/app/(admin)` 라우트는 **운영자(서비스M) 슈퍼어드민**이다. 이 요구서의 병원 어드민과 **다른 사용자**다.
- 병원 어드민은 신규. 배치 위치(라우트 그룹·별도 앱 등)·격리 구현 방식은 **개발자 재량**. 단 §6 비기능 요구의 병원 간 격리는 필수.
- 재사용 가능 자산: `Clinic`·`Booking`·`User` 모델, better-auth, Gemini 번역(`@google/generative-ai`), S3 이미지 업로드, 기존 `(service)` 클리닉 페이지 렌더링 로직.

### 1.4 우선순위 표기

- **P1**: MVP 필수
- **P2**: 후속 확장 (이번 1차 구현 범위 밖, 문서엔 표기만)

## ⚠️ 개발 선결 과제 (현재 스키마로 바로 구현 불가 — 우선 처리)

아래 두 가지는 기존 `Booking`/`Clinic` 스키마에 없어, **F3·F5 구현 전 스키마 변경이 선행**되어야 한다.

### S1. 예약 상태값 확장 (F3 전제)
- 현재 `Booking.status` = **`PENDING / CONFIRMED / CANCELED` 3개뿐**.
- F3의 6단계(접수→확정→내원→상담완료→시술완료→사후관리) + 노쇼/취소를 담으려면 **status 값 확장(enum화 권장)** 필요.
- 특히 **시술완료** 상태가 없으면 F5 수수료 트리거가 동작하지 않는다.
- 상태 변경 이력(누가·언제) 테이블도 신규 필요(§5.2).

### S2. 시술 금액·플랜 데이터 출처 (F5 전제)
- 현재 `Booking`에 **결제액·시술가 필드 없음**. 구글 피드 가격도 `INTERPRETATION_NOT_DISPLAYED`이고, `PaymentSession` 모델은 예약(`Booking`)과 연결돼 있지 않다.
- F5의 "건별 결제액 × 수수료율" 계산을 위해 **시술 금액 입력 경로**(병원이 시술완료 시 금액 입력, 또는 별도 금액 필드·모델)를 정의해야 한다.
- 병원 **플랜(베이직/스탠다드/프리미엄)** 저장 위치도 필요 — 현재 `Clinic`에 플랜 필드 없음. 수수료율(20%/15%) 자동 적용의 전제다.

## 3. F1 — 로그인·계정

### 동작

- 병원 계정으로 로그인 (better-auth 기반).
- 로그인 후 자기 병원(`clinicId`)에 귀속된 데이터만 노출.
- 비밀번호 재설정.

### 데이터

- 병원 계정 ↔ `Clinic` 매핑 (계정 1개당 병원 1곳).

### 수용 기준

- A 병원 계정으로 B 병원의 예약·환자·페이지에 **어떤 경로로도 접근 불가** (URL 직접 입력 포함).

## 4. F2 — 대시보드

로그인 직후 첫 화면. 한눈에 보는 현황 요약.

### 표시 항목

- **오늘 예약** 건수 + 목록 바로가기
- **신규 환자**(미확인 신규 예약) 건수 — F3로 연결
- **이번 달 시술완료** 건수
- **이번 달 예상 수수료** 합계 (F5 요약, 읽기전용)
- **미응답 채팅** 건수 (F4 연결)

### 수용 기준

- 모든 수치는 로그인 병원 데이터로만 집계.
- 각 카드 클릭 시 해당 모듈(F3/F4/F5)로 이동.

## 5. F3 — 예약·환자 관리 (메인)

이 어드민의 핵심. 들어온 예약을 통합 수신하고 진행상태를 관리한다.

### 5.1 예약 인박스

- **예약은 단일 경로로 수신된다**: 모든 예약은 자체 booking 폼(`createBooking()` → `Booking` 테이블) **한 곳**으로 적재된다.
- 구글 지도 "Book Online"은 별도 시스템이 아니다 — Google Actions Center(Reserve with Google) **action feed**(기존 `api/cron/feed_create`·`feed_upload`, `isGoogleEnabled:true` 클리닉만 대상)에 등록된 링크가 이 **자체 booking 폼으로 redirect**되는 구조다. 따라서 어드민은 `Booking` **한 소스만** 표시하면 된다.
- 유입 경로(구글 지도 경유 vs 사이트 직접)를 구분 표시하려면 booking 진입에 출처 파라미터가 필요하다 — **현재 `Booking` 모델에 출처 필드가 있는지 개발자 확인 필요**. 있으면 배지/필터로 노출, 없으면 P2.
- 정렬·필터: 상태별, 날짜별, 검색(여권명·전화·이메일).

### 5.2 예약 상태 라이프사이클 (6단계 + 예외)

```
접수 → 확정 → 내원 → 상담완료 → 시술완료 → 사후관리
                                  └ (시술완료 = 수수료 정산 트리거)
예외 상태: 노쇼 / 취소  (어느 단계에서든 전환 가능)
```

- 병원이 각 건의 상태를 수동 전환.
- **시술완료** 전환 시 F5 수수료 집계에 자동 반영.
- 상태 변경 이력(누가·언제) 기록.
- ⚠ **선결**: 현재 `status`는 3개 값뿐 → §개발 선결 과제 **S1** 처리 후 구현 가능.

### 5.3 환자 카드 (예약 상세)

- 환자 정보(기존 `Booking` 필드 기준): 여권명(`passportName`), 국적(`nationality`), 출발국(`startNation`), 성별·생년(`gender`·`birthDate`), 연락처(`phoneNumber`·`email`), 메신저(`messengerType`·`messengerId`), 요청 메시지(`message`).
- 예약 정보: 희망일시(`visitDate`·`visitTime`), 선택 시술(`bookingService`), 현재 상태.
- **방문/시술 이력**: 동일 환자의 과거 예약 목록 (재방문 식별).
- **내부 메모**: 병원 내부용 자유 메모 (환자에게 비노출).
- 환자 카드에서 F4 채팅 바로 열기.

### 데이터 (재사용/확장)

- 기존 `Booking`·`User`(환자) 모델 활용. 상태 enum·이력 테이블·출처 필드는 신규 가능.

### 수용 기준

- 구글 예약과 자체폼 건이 **하나의 목록**에서 동일하게 다뤄진다.
- 상태 전환이 즉시 반영되고, 시술완료 시 F5에 자동 집계된다.

---

## 6. F4 — 환자 다국어 채팅

병원이 외국인 환자와 직접 소통. 언어 장벽을 자동번역으로 해소.

### P1 (필수)

- 예약/환자 단위 1:1 텍스트 채팅.
- **자동번역**: 환자 언어 ↔ 한국어 양방향 (Gemini 재사용). 병원은 한국어로 입력·열람, 환자는 자기 언어로.
- 원문/번역문 함께 표시(번역 오류 대비).
- 미응답 표시 (F2 대시보드 연동).

### P2 (확장 — 이번 범위 밖, 표기만)

- 파일·이미지 첨부
- 푸시/이메일/메신저 알림
- 정형 답변 템플릿

### 수용 기준

- 병원이 한국어만으로 외국어 환자와 메시지를 주고받을 수 있다.
- 채팅은 해당 병원-환자 간에만 보인다.

---

## 7. F5 — 수수료 현황 (읽기전용)

투명성 목적. 병원이 자기 정산 예정액을 직접 확인. **청구·결제 기능 아님.**

### 표시 항목

- 기간(월) 선택.
- **시술완료** 건 목록 + 건별 결제액 + 적용 수수료율.
- 수수료율 규칙: **베이직 20% / 스탠다드·프리미엄 15%** (병원 플랜에 따라 자동 적용).
- 기간 합계 **예상 수수료**.
- ⚠ **선결**: `Booking`에 금액 필드 없음 + `Clinic`에 플랜 필드 없음 → §개발 선결 과제 **S2** 처리 후 구현 가능.

### 수용 기준

- F3에서 시술완료된 건만 집계된다.
- 화면은 **읽기 전용** — 병원이 금액을 수정하거나 청구를 발생시킬 수 없다.

---

## 8. F6 — 공개 페이지 편집기

병원이 자기 공개 클리닉 페이지를 직접 수정.
(예: `서비스M.com/en/clinic/c-clinic-dermatology-dongdaemun`)

### 8.1 편집 항목 (↔ 기존 `Clinic` 스키마 매핑, 2026-06-07 확인)

| 화면 항목         | Clinic 필드                                                              | 비고                                                              |
| ----------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------- |
| 대표 사진         | `image`                                                                | S3 업로드                                                         |
| 갤러리(인테리어)  | `interiorImages[]`                                                     |                                                                   |
| 로고              | **(필드 없음)**                                                    | 현재 Clinic에 로고 필드 없음 → 필요하면 신규 필드, 아니면 미지원 |
| 소개글            | `description`                                                          |                                                                   |
| 클리닉명·지점명  | `name`·`branchName`                                                 | ⚠ 구글 피드·`slug` 영향 (§8.5)                               |
| 진료과            | `category`                                                             | 'derm'·'plastic' 등                                              |
| 주소              | `address`·`locality`·`region`·`streetAddress`·`postalcode` | **다중 필드** (단일 입력 아님)                              |
| 지도 좌표         | `latitude`·`longitude`                                              | ⚠ 구글 피드 사용 (§8.5)                                         |
| 진료시간          | `hours[]` (ClinicHour 관계)                                            | 요일별                                                            |
| 브레이크/접수마감 | `breakTime`·`reserveClose`                                          |                                                                   |
| 서비스 태그       | `tags[]`                                                               |                                                                   |
| 클리닉 특징       | `features[]`                                                           |                                                                   |
| 핵심 강점         | `keyStrengths[]`                                                       |                                                                   |
| 공지/안내         | `notices[]`                                                            | 공개 페이지 안내문구                                              |
| 인증 배지         | `badges[]`                                                             | ISO/JCI 등 — 허위표시 방지 위해 운영자 검수 권장                 |
| 시그니처 시술     | `signatureTreatments[]`                                                |                                                                   |
| 시술 카테고리     | `TreatmentCategories` (Json)                                           | 구조화 데이터                                                     |
| 예약 서비스       | `bookingServices[]`                                                    | ⚠ 구글 action feed에 직접 사용 (§8.5)                           |
| 의사 프로필       | `doctors[]` (Doctor 관계)                                              | 사진·이름·약력                                                  |
| 통역 가능 언어    | `languages[]`                                                          |                                                                   |
| 병원 홈페이지 URL | `webpageUrl`                                                           |                                                                   |

> `rating`·`reviewCount`·`likes`는 자동 집계 통계 — 편집 대상 아님(§8.4).

### 8.2 다국어 처리

- 병원은 **한국어로 입력** → Gemini 자동번역으로 다국어 페이지(en·ja·zh 등) 생성.
- 언어별 **수동 보정** 가능 (자동번역 위에 덮어쓰기).

### 8.3 공개 반영

- 저장 시 공개 페이지에 반영.
- **운영자 검수 게이트**(저장 → 승인 → 공개)는 **옵션** — 개발자/운영 정책에 따라 on/off 가능하게 설계 권장.

### 8.4 리뷰

- 환자 리뷰는 병원 **열람만** (작성은 환자 영역).
- 리뷰 응답 기능은 **P2**.

### 8.5 구글 Reserve 피드 연동 필드 — 편집 제한 (중요)

아래 필드는 Google Actions Center entity/action feed(`api/cron/feed_create`·`feed_upload`)에 **직접 반영**된다. 병원이 자유 편집하면 구글 지도 노출·예약 링크 정합성이 깨질 수 있다:

- `isGoogleEnabled`, `googleStreetAddress`, `latitude`/`longitude`, `bookingServices[]`, `name`/`branchName`/`slug`

→ 이 필드들은 **병원 편집에서 제외(운영자 전용)** 하거나, 편집 시 **운영자 승인 게이트(§8.3)를 강제**한다. 어느 쪽으로 갈지는 운영 정책 결정 필요.

- 참고: `googleStreetAddress`는 구글 피드 전용 주소 오버라이드 필드 — 병원 표시용 주소(`address` 등)와 분리 운영된다.

### 수용 기준

- 병원이 한국어 입력만으로 다국어 공개 페이지를 갱신할 수 있다.
- 이미지 업로드는 기존 S3 presigned-url 흐름 재사용.

## 9. 비기능 요구사항

- **병원 간 데이터 격리 (필수)**: 모든 조회·수정 요청이 로그인 병원의 `clinicId`로 스코프된다. 타 병원 데이터 노출 0건. (F1 수용 기준과 동일 원칙을 전 모듈 적용)
- **다국어 UI**: 어드민 UI 언어는 한국어 우선. 환자 대상 콘텐츠(F4·F6)는 다국어.
- **개인정보·의료정보 취급**: 환자 이름·연락처·국적·예약 시술 등 민감정보 포함. 접근 통제·전송 보안 준수.
- **모바일 대응**: 병원 실무자가 모바일에서도 예약 확인·상태 변경·채팅 가능.

## 10. 범위 밖 (Out of Scope)

- 실제 수수료 **청구서 발행·결제·정산 처리** (F5는 표시만)
- 운영자(서비스M) 슈퍼어드민 기능 (기존 `(admin)` 영역)
- Google Reserve **피드 생성/업로드** (기존 `api/cron/feed_*` 존재)
- 병원 내 역할/권한 세분화 (원장·실장 차등)

## 11. 검수 중 발견된 별도 이슈 (요구서 범위 밖, 확인 요망)

- **예약완료 페이지에 경쟁사 브랜드 노출** — `apps/web/app/(service)/[lang]/booking/[id]/complete/page.tsx` 헤더에 `[경쟁사 브랜드]` 로고/텍스트가 하드코딩돼 있다. 외국인 환자가 예약 완료 시 보는 화면이므로 서비스M 브랜드로 교체 필요. (템플릿 복사 흔적으로 추정)
