# 우리 회사 레드팀 설계 스펙

> 비즈니스 문제를 주기적으로 발굴하고, 아이젠하워 매트릭스로 분류하여 보고하는 자동화 시스템

## 1. 목적

우리 회사의 6개 데이터 소스를 주기적으로 분석하여 비즈니스 문제점과 개선 기회를 발굴한다. 기존 업무를 방해하지 않으면서, 발견한 이슈를 우선순위별로 분류하여 의사결정자에게 전달한다.

### 핵심 제약

- 기존 업무 프로세스를 방해하지 않는다
- 즉시 경보는 긴급+중요(🔴)만 허용한다
- 모든 액션은 사용자 승인 후 실행한다
- 기존 시스템(오케스트레이터, 파이프라인)을 수정하지 않는다

## 2. 시스템 구조

### 2.1 에이전트 구성

두 개의 독립적인 Claude Code schedule 에이전트로 구성한다.

| 에이전트 | 실행 주기 | 역할 |
|---------|----------|------|
| 주간 레드팀 | 매주 월요일 09:00 | 6개 소스 전체 스캔, 아이젠하워 4분류 보고서 생성 |
| 일일 워치독 | 매일 08:00 | 핵심 지표 이상치 체크, 🔴만 즉시 경보 |

### 2.2 데이터 흐름

```
데이터 소스 (6개)
    ↓
데이터 수집 레이어
    ↓
분석 엔진 (전주 대비, 추세 이탈, 기준선 비교)
    ↓
아이젠하워 분류
    ↓
┌─────────────┬──────────────┬─────────────┐
│ iMessage 보고 │ 노션 기록     │ 로컬 기록    │
│ (요약+분류)   │ (개인OS)    │ (JSON)      │
└─────────────┴──────────────┴─────────────┘
```

## 3. 데이터 소스 및 수집 항목

### 3.1 재무 (Granter API)

- **수집**: 주간 매출, 비용, 잔액
- **기준선**: 최근 4주 이동평균
- **초기 트리거**: 매출 n%↓, 비용 n%↑, 잔액 n만원 이하

### 3.2 리드 (Meta 광고 + 진단 폼)

- **수집**: 주간 리드 수, CPL, 소스별 비중
- **기준선**: 최근 4주 이동평균
- **초기 트리거**: 리드 수 n%↓, CPL n%↑

### 3.3 고객 (온톨로지)

- **수집**: 계약 현황, 유지율, 이탈 징후 (계약 만료 30일 이내)
- **초기 트리거**: 계약 만료 임박 고객 미갱신, 동시 2건 이상 이탈

### 3.4 마케팅 (Meta 광고 + pluuug CRM)

- **수집**: Meta 광고 성과 (광고비, CPL, 전환율, ROAS) + pluuug 파이프라인 데이터 (리드 단계별 건수, 전환율, 체류일)
- **기준선**: 최근 4주 이동평균
- **초기 트리거**: 전환율 n%↓, 예산 소진율 비정상, 파이프라인 병목 (특정 단계 체류일 급증)
- **선행 작업**: pluuug API 연동 (`openapi.pluuug.com/v1`, API 키 `.env`에 저장 완료)

### 3.5 운영 (업무 로그)

- **수집**: daily-automation.py가 생성하는 업무일지 데이터, 온톨로지 operations/ 문서
- **분석**: 반복 패턴 식별 및 자동화 가능 작업 태깅 → 🔵로 분류
- **참고**: 정형 데이터가 아니므로 정성적 분석 위주. 정량 트리거 없음

### 3.6 GBP 대시보드

- **수집**: 병원별 랭킹 변동, 리뷰 수 변화
- **초기 트리거**: 주요 고객 랭킹 5위 이상 하락

## 4. 아이젠하워 분류 체계

| 분류 | 정의 | 보고 방식 |
|------|------|----------|
| 🔴 긴급+중요 | 매출/리드에 즉시 영향, 방치하면 손실 | iMessage 즉시 경보 |
| 🟡 중요+비긴급 | 개선하면 성장에 기여, 당장 손실 없음 | 주간 보고서에 포함 |
| 🔵 긴급+비중요 | 빨리 처리해야 하지만 직접 할 필요 없음 | 주간 보고서 + 자동화 제안(🤖) |
| ⚪ 비긴급+비중요 | 알아두면 좋지만 액션 불필요 | 로컬/노션에 기록만 |

### 분류 규칙

- 🔴만 즉시 iMessage 발송. 나머지는 주간 보고서에 모아서 전달
- 🔵에는 Claude Code 자동화 가능 여부를 🤖로 태깅
- 한 주에 🔴이 없으면 "이번 주 긴급 이슈 없음" 한 줄로 끝
- 🤖 태깅된 항목은 사용자가 "이거 해"라고 하면 개발 착수

## 5. 보고 형식

### 5.1 iMessage 주간 보고 (월요일 09:00)

```
📊 우리 회사 레드팀 주간 리포트 (3/24~3/30)

🔴 긴급+중요
  없음

🟡 중요+비긴급
  1. CPL n원 → n원 (3주 연속 상승)
     → 광고 소재 교체 검토
  2. A병원 계약 만료 D-25, 갱신 논의 미시작
     → 이번 주 연락 권장

🔵 긴급+비중요 (자동화 가능 🤖 표시)
  1. 🤖 주간 고객 리포트 수동 작성 (주 n시간 소요)
     → Claude Code 자동화 가능
  2. 진단 폼 리드 중복 수동 필터링
     → 웹훅 단에서 자동 처리 가능 🤖

⚪ 참고
  1. 경쟁사 B, 구글 광고 시작한 듯 (GBP 변동)

📈 핵심 지표 요약
  매출: n만원 (전주 대비 -3%)
  리드: n건 (전주 대비 +2건)
  잔액: n만원
  파이프라인: 상담 n → 제안 n → 계약 n
```

### 5.2 즉시 경보 (🔴 트리거 발동 시)

```
🚨 레드팀 경보
리드 유입 주간 n% 급감 (n건→n건)
원인 추정: Meta 광고 노출 감소
권장 액션: 광고 관리자 확인
```

짧고 한 건씩만. 판단은 사용자에게.

### 5.3 노션 기록 (개인OS)

- 주간 보고서를 노션 페이지로 생성
- 이슈별 상태 추적: 발견 → 조치중 → 해결
- 과거 보고서 누적으로 트렌드 확인 가능

### 5.4 로컬 기록

```
company-pipeline/redteam/
├── reports/
│   └── 2026-W13-report.json    # 주간 보고 원본 데이터
├── alerts/
│   └── 2026-03-28-lead-drop.json  # 트리거 경보 이력
├── config.json                  # 임계값 설정
└── baseline.json                # 기준선 데이터 (4주 이동평균)
```

## 6. 임계값 학습 구조

### 6.1 피드백 루프

| 사용자 반응 | 시스템 동작 |
|------------|-----------|
| "이건 너무 민감해" | 해당 임계값 10%p 완화 |
| "이건 놓쳤어" / "왜 안 알려줬어" | 해당 임계값 10%p 강화 |
| "이건 🔴이 아니라 🟡야" | 분류 기준 조정 기록 |
| "이건 안 봐도 돼" | 해당 항목 감시 제외 |
| 무반응 | 현재 설정 유지 |

**피드백 수신 방식**: iMessage로 보고서에 대한 반응이 오면 해당 세션의 에이전트가 config.json을 업데이트한다. 스케줄 에이전트 실행 간에는 피드백이 반영되지 않으므로, 피드백은 다음 주간 보고 시 반영된다.

### 6.2 config.json 구조

```json
{
  "thresholds": {
    "finance": {
      "revenue_drop": { "value": "n", "unit": "%", "adjusted": false },
      "cost_spike": { "value": "n", "unit": "%", "adjusted": false },
      "balance_min": { "value": "n", "unit": "won", "adjusted": false }
    },
    "leads": {
      "count_drop": { "value": "n", "unit": "%", "adjusted": false },
      "cpl_spike": { "value": "n", "unit": "%", "adjusted": false }
    },
    "clients": {
      "contract_expiry_days": { "value": 30, "unit": "days", "adjusted": false },
      "simultaneous_churn": { "value": 2, "unit": "count", "adjusted": false }
    },
    "marketing": {
      "conversion_drop": { "value": "n", "unit": "%", "adjusted": false },
      "pipeline_stall_days": { "value": 14, "unit": "days", "adjusted": false }

```

- `adjusted: true`는 사용자 피드백으로 튜닝된 값
- `feedback_history`에 변경 이력 기록
- 4주 뒤 조정된 값이 오탐/미탐 없으면 안정화 처리

## 7. 구현 방식

Claude Code `schedule` 기능으로 두 개의 크론 에이전트를 등록한다.

### 7.1 주간 레드팀 에이전트

- **스케줄**: 매주 월요일 09:00
- **동작**: 6개 데이터 소스 수집 → 기준선 대비 분석 → 아이젠하워 분류 → iMessage 보고 + 노션 기록 + 로컬 저장
- **입력**: config.json (임계값), baseline.json (기준선)
- **출력**: 주간 보고서 (iMessage + 노션 + JSON)

### 7.2 일일 워치독 에이전트

- **스케줄**: 매일 08:00
- **동작**: 핵심 지표(재무 잔액, 리드 수, 고객 이탈)만 빠르게 체크
- **출력**: 이상치 발견 시에만 🔴 경보 iMessage 발송. 정상이면 무출력

### 7.3 선행 작업

- pluuug API 연동 (`openapi.pluuug.com/v1`, API 키 `.env` 저장 완료)
- pluuug API 스펙 조사 및 데이터 수집 함수 구현
- Granter API 연동 확인 (API 키 `.env` 저장 완료, daily-automation.py에서 이미 사용 중)

### 7.4 콜드 스타트 정책

- 최초 실행 시 4주치 기준선 데이터가 없으므로, 처음 2주는 데이터 수집만 수행하고 경보를 발송하지 않는다
- 2주차부터 2주 이동평균으로 임시 기준선을 사용한다
- 4주차부터 정상 운영 (4주 이동평균 기준선)

## 8. 기존 업무 보호 원칙

1. 🔴 아닌 것은 절대 즉시 알림을 보내지 않는다
2. 주간 보고서도 핵심만 — 길어지면 자르고 상세는 노션 링크로
3. 🔵 자동화 제안은 태깅만, 실행은 사용자 승인 후
4. 기존 오케스트레이터, 파이프라인 코드를 수정하지 않는다
5. 새로운 업무를 만들어내지 않는다 — 이미 존재하는 문제를 가시화할 뿐

## 9. 성공 기준

- 주간 보고서가 매주 월요일 09:00에 iMessage로 도착한다
- 🔴 트리거 발동 시 당일 내 iMessage 경보가 온다
- 노션과 로컬에 보고 이력이 누적된다
- 임계값 피드백이 config.json에 반영된다
- 🔴 외 즉시 경보 발송 0건/주 (기존 업무 비방해 객관적 지표)
