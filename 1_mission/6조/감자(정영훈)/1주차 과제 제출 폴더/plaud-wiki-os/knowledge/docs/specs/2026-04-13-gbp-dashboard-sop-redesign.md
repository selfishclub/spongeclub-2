# GBP 대시보드 SOP 재설계 — 운영 중심 대시보드

## 개요

기존 GBP 대시보드를 **데이터 뷰어 → 운영 관리 도구**로 전면 재설계. "열면 바로 뭐 해야 하는지 보이는" 대시보드로 전환.

**핵심 문제:**
- 리포트가 히스토리로 쌓이지 않음
- 성과(순위/리뷰/조회수) 시간축 변화를 추적할 수 없음
- 관리 품질(리뷰 답글 빠뜨림, 업데이트 주기 미준수)을 모니터링할 수 없음

**해결 방향:**
- SOP 체크리스트 기반 관리 품질 모니터링
- 거래처별 건강 점수 (100점 만점)
- 액션 시스템으로 할 일 자동 생성
- 성과 트렌드 시계열 추적
- 리포트 히스토리 축적

**사용자:** 본인 (우리 회사 대표)
**플랫폼:** 기존 gbp-dashboard (Next.js 15 + Supabase)

## 정보 구조

### 메인 화면 (홈)

대시보드를 열면 위에서 아래로 3단 구성:

```
┌─────────────────────────────────────────────┐
│  1. 액션 리스트                               │
│  긴급(🔴) → 주의(🟡) → 예정(🔵) → 제안(⚪)   │
├─────────────────────────────────────────────┤
│  2. 거래처 건강 점수 카드 (N개)               │
│  카드마다: 이름 + 점수 + 증감 + 색상           │
├─────────────────────────────────────────────┤
│  3. 주간 성과 요약                            │
│  평균 순위 | 총 리뷰 증감 | 관리 점수 평균      │
└─────────────────────────────────────────────┘
```

### 거래처 상세 (카드 클릭)

```
┌─────────────────────────────────────────────┐
│  [거래처명] 건강 점수 N/100                    │
├──────────────┬──────────────────────────────┤
│  SOP 체크리스트 │  성과 트렌드 그래프           │
│  (초기 세팅     │  (순위/리뷰/건강점수 추이)     │
│   + 반복 운영)  │                             │
│               ├──────────────────────────────┤
│               │  리포트 히스토리               │
│               │  (열기 / 재전송)               │
└──────────────┴──────────────────────────────┘
```

기존 기능(리뷰 목록, 순위 조회, 경쟁사 분석 등)은 거래처 상세 안 탭으로 이동.

## 건강 점수 시스템

### 배점 구조 (100점 만점)

#### 초기 세팅 (30점) — 1회 완료 후 유지

| 항목 | 배점 | 감지 방식 |
|------|------|----------|
| 키워드 선정 완료 | 8점 | 수동 |
| GBP 기본정보 세팅 | 7점 | 수동 |
| 카테고리 5개+ 설정 | 5점 | 수동 |
| 사진 배치 완료 | 5점 | 수동 |
| AEO 스키마 반영 | 5점 | 수동 |

#### 반복 운영 (70점) — 매주 자동 갱신

| 항목 | 배점 | 감지 방식 |
|------|------|----------|
| 리뷰 답글 24h 이내 | 25점 | **자동** (답글 시간 - 리뷰 시간) |
| 업데이트 주 2개+/언어 | 20점 | **자동** (게시물 감지) |
| 키워드 포함률 (답글/업데이트) | 15점 | **자동** (텍스트에 타겟 키워드 존재 여부) |
| Q&A 관리 | 10점 | 수동 |

### 점수 계산

- 반복 운영 점수는 **최근 7일** 기준으로 매일 재계산
- 리뷰 답글: (24h 이내 답변 수 / 전체 신규 리뷰 수) × 25점
- 업데이트: 언어별 2개 이상 충족 시 만점, 미달 시 비례 감점
- 키워드 포함률: (키워드 포함 답글+업데이트 수 / 전체 수) × 15점
- 리뷰가 0건인 주는 리뷰 답글 항목 만점 처리 (감점 없음)

### 색상 기준

| 점수 | 색상 | 의미 |
|------|------|------|
| 90~100 | 🟢 초록 | 잘 관리 중 |
| 70~89 | 🟡 노랑 | 신경 써야 함 |
| 0~69 | 🔴 빨강 | 방치 중 |

## 액션 시스템

### 액션 타입

| 타입 | 긴급도 | 예시 | 감지 |
|------|--------|------|------|
| 🔴 긴급 | 즉시 | "협조병원1 리뷰 2건 미답변 (20h 경과)" | 자동 |
| 🟡 주의 | 오늘 중 | "협조병원2 이번주 업데이트 0/2" | 자동 |
| 🔵 예정 | 이번주 | "C의원 사진 리뉴얼 필요" | 수동 등록 |
| ⚪ 제안 | 참고 | "협조병원3 Q&A 3개 미응답" | 자동 |

### 자동 생성 규칙 (매일 Cron)

1. **미답변 리뷰 스캔** → 경과 시간 기준 🔴(18h+) 또는 🟡(12h+) 생성
2. **주간 업데이트 수 집계** → 부족 시 🟡 생성
3. **키워드 포함률 체크** → 최근 답글/업데이트에 타겟 키워드 누락 패턴 발견 시 ⚪ 생성
4. **초기 세팅 미완료 항목** → 🔵 생성

### 처리 흐름

```
액션 카드 클릭
  → 해당 거래처 상세로 이동
  → 관련 SOP 항목 하이라이트
  → 처리 후 자동 또는 수동으로 완료 체크
  → 건강 점수 즉시 재계산
```

### 표시 규칙

- 긴급도 순 정렬 (🔴 → 🟡 → 🔵 → ⚪)
- 🔴, 🟡는 항상 표시. ⚪는 접혀 있고 "N건 더 보기"로 펼침
- 0건이면: "모든 거래처 관리 정상 🟢" 한 줄 표시

## 성과 트렌드

### 메인 화면 — 주간 성과 요약

전체 거래처 집계:
- 평균 순위 + 전주 대비 변동
- 총 리뷰 증감 + 답글 완료율
- 관리 점수 평균 + 전주 대비 변동

### 거래처 상세 — 성과 그래프

기간 선택 (최근 4주 / 8주 / 12주):
- **순위 추이**: 키워드별 라인 차트
- **리뷰 추이**: 신규 리뷰 수, 평균 별점, 평균 응답 시간
- **건강 점수 추이**: 주간 점수 라인 차트

### 데이터 수집

- 매일 Cron에서 순위/리뷰/건강 점수를 스냅샷으로 저장
- 기존 DataForSEO 스캔 데이터 + 리뷰 싱크 데이터 활용
- 신규: `health_score_snapshots` 테이블에 일일 점수 기록

## 리포트 히스토리

### 기존 문제

리포트를 생성해서 텔레그램으로 보내지만, 대시보드에는 기록이 남지 않음.

### 해결

- 리포트 생성 시 `reports` 테이블에 메타데이터 + 파일 경로 저장
- 거래처 상세에서 리포트 목록 조회 (생성일, 유형, 열기/재전송)
- 기존 PPT 생성 파이프라인 출력 단계에 DB INSERT 추가

## 데이터 모델 (Supabase 추가/수정)

> **참고:** 기존 스키마에서 FK 컬럼명은 `client_id`, PK 테이블은 `clients`. 아래 모든 신규 테이블은 이 명명 규칙을 따른다.

### `sop_checklist` (신규)
거래처별 SOP 초기 세팅 + 주간 반복 체크 상태.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| client_id | uuid | FK → clients |
| item_key | text | 'keyword_setup', 'basic_info', 'categories', 'photos', 'aeo_schema', 'qa_weekly' |
| completed | boolean | 완료 여부 |
| completed_at | timestamptz | 완료 시각 |
| week_of | date | NULL = 초기 세팅(영구), 값 있으면 해당 주 월요일 (주간 리셋 항목용) |

- UNIQUE 제약: `(client_id, item_key)` (초기 세팅), `(client_id, item_key, week_of)` (주간 항목)
- Q&A 관리(`qa_weekly`)는 매주 월요일 Cron이 새 행 INSERT (week_of = 해당 주 월요일)

### `health_score_snapshots` (신규)
일일 건강 점수 스냅샷.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| client_id | uuid | FK → clients |
| date | date | 스냅샷 날짜 |
| setup_score | smallint | 초기 세팅 점수 (0~30) |
| review_score | smallint | 리뷰 답글 점수 (0~25) |
| update_score | smallint | 업데이트 점수 (0~20) |
| keyword_score | smallint | 키워드 포함률 점수 (0~15) |
| qa_score | smallint | Q&A 점수 (0~10) |
| total_score | smallint | 합계 (0~100) |

- UNIQUE 제약: `(client_id, date)` — 같은 날 중복 삽입 방지 (Cron은 UPSERT 사용)

### `actions` (신규)
자동/수동 생성 액션.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| client_id | uuid | FK → clients |
| type | text | 'urgent', 'warning', 'scheduled', 'suggestion' |
| action_key | text | 중복 방지 키 (예: 'review_overdue__{review_id}', 'update_shortage__{week}') |
| title | text | 액션 제목 |
| description | text | 상세 설명 |
| source | text | 'auto' or 'manual' |
| resolved | boolean | 처리 완료 여부 (default: false) |
| created_at | timestamptz | 생성 시각 |
| resolved_at | timestamptz | 처리 시각 |

- UNIQUE 제약: `(client_id, action_key)` WHERE `resolved = false` — 미처리 액션 중복 방지
- Cron은 INSERT ... ON CONFLICT DO NOTHING 패턴 사용

### `reports` (기존 테이블 수정)
기존 `reports` 테이블에 `sent_at` 컬럼만 추가. 신규 테이블 생성하지 않음.

| 추가 컬럼 | 타입 | 설명 |
|----------|------|------|
| sent_at | timestamptz | 전송 시각 (텔레그램), NULL이면 미전송 |

### 순위 데이터 — 기존 `local_rankings` 활용
기존 `local_rankings` 테이블이 이미 `(client_id, keyword, scan_date, avg_rank, grid_data)`를 저장하고 있으므로 별도 `ranking_snapshots` 테이블을 만들지 않음. 성과 트렌드 그래프는 `local_rankings`에서 직접 쿼리.

## 감지 방식 상세

### 자동 감지 (GBP API / 기존 Cron 활용)

| 항목 | 데이터 소스 | 감지 방법 |
|------|-----------|----------|
| 리뷰 답글 24h | 기존 리뷰 싱크 Cron | 리뷰 생성 시각 - 답글 생성 시각 비교 |
