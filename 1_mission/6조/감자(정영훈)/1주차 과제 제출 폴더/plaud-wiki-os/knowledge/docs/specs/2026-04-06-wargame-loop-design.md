# Wargame Loop + Demo Preview 설계 스펙

## 개요

비개발자가 기능 구현 후 오류 가능성을 최소화하기 위한 자동 검수 루프 시스템.
구현 결과물을 Chrome 원격 데스크톱으로 바로 확인할 수 있는 데모 프리뷰 통합.

## 배경

- 사용자는 비개발자로, 에러를 직접 잡아낼 수 없음
- Mac이 사무실에 24시간 가동, Chrome 원격 데스크톱으로 접속
- 기존 에이전트(code-reviewer, e2e-runner, build-error-resolver 등)는 있으나 수동 호출 필요
- "전부 맡겨" 자율 모드(Step 4-A)에 검수가 빠져있음

## 시스템 구성

### 산출물

| 파일 | 역할 |
|------|------|
| `skills/wargame/SKILL.md` | 검수 루프 스킬 본체 |
| `skills/demo-preview/SKILL.md` | 데모 프리뷰 스킬 |
| `reports/wargame/wargame-lessons.md` | 반복 실수 패턴 DB (자동 누적) |
| `CLAUDE.md` 수정 | 단축어 + Step 4-A 확장 |
| `reports/wargame/` | 리포트 저장 디렉토리 |

## 1. 검수 루프 (Wargame)

### 트리거

| 상황 | 트리거 | 동작 |
|------|--------|------|
| "검수해" / "워게임" | 수동 | 현재 프로젝트에 검수 루프 시작 |
| "전부 맡겨" 구현 완료 | 자동 | Step 4-A 마지막에 워게임 자동 진입 |
| "데모 띄워" | 수동 | 검수 없이 프리뷰만 |

### 이슈 심각도 분류

| 등급 | 정의 | PASS 차단 | 예시 |
|------|------|-----------|------|
| CRITICAL | 보안 취약점, 데이터 손실 위험 | 반드시 차단 | XSS, SQL injection, 비밀키 노출 |
| HIGH | 런타임 에러, 기능 미동작 | 차단 | 빌드 실패, API 에러, E2E 실패 |
| MEDIUM | 코드 품질, 잠재적 문제 | 차단 안 함 (리포트 기록) | 미사용 import, 타입 누락, 관용구 위반 |
| LOW | 스타일, 사소한 개선 | 차단 안 함 (리포트 기록) | 네이밍, 포맷팅, 주석 |

**PASS 조건:** CRITICAL + HIGH 이슈가 0건. MEDIUM/LOW는 리포트에 남기되 PASS 판정에 영향 없음.

### 루프 구조

```
/wargame 트리거
  ↓
Round N (최대 20):
  ├─ Phase 1: 빌드 검증
  │    프로젝트 타입 자동 감지 (Next.js/Python/Shell 등)
  │    빌드/린트 실행 → 실패 시 자동 수정 후 재빌드
  │
  ├─ Phase 2: 코드 품질 (병렬 실행)
  │    ① code-reviewer → 구조/품질 이슈
  │    ② security-reviewer → 보안 취약점
  │    ③ 언어별 reviewer (python/ts/등) → 언어 관용구
  │    ④ wargame-lessons.md 반복 패턴 우선 체크
  │    → 각 이슈에 CRITICAL/HIGH/MEDIUM/LOW 등급 부여
  │
  ├─ Phase 3: E2E 테스트 (웹 프로젝트만)
  │    e2e-runner로 브라우저 테스트
  │    스크린샷 캡처 → 리포트에 포함
  │    (스크립트/파이프라인은 Phase 3 스킵)
  │
  ├─ 이슈 수집 & 자동 수정
  │    CRITICAL/HIGH 이슈만 자동 수정 (최소 변경 원칙)
  │    MEDIUM/LOW는 리포트 기록만
  │    수정 내역 리포트에 기록
  │
  └─ 판정
       CRITICAL+HIGH 0건 → 조기 종료, PASS
       CRITICAL+HIGH 남음 → Round N+1
       Round 20 도달 → 강제 종료, FAIL
```

### 조기 종료 조건

- CRITICAL + HIGH 이슈 0건 (MEDIUM/LOW 잔존 허용)
- 연속 2라운드 동일 오류 (수정 불가능 판단) → FAIL + 사용자에게 보고

### 토큰 예산 관리

- 라운드당 서브에이전트 최대 4개 (Phase 2 병렬 3개 + Phase 3 E2E 1개)
- **컨텍스트 60% 도달 시:** 현재 라운드까지 리포트 저장 + 커밋 후 조기 종료
- 조기 종료 시 커밋: `wargame: STOPPED round N/20 — context limit, 잔여 M건`
- 사용자에게 "새 세션에서 /wargame으로 재개 가능" 안내

### 커밋 컨벤션

- 성공: `wargame: PASS round 3/20 — 수정 4건`
- 실패: `wargame: FAIL round 20/20 — 잔여 2건`


## 3. 데모 프리뷰

### 프로젝트 타입별 동작

| 타입 | 감지 기준 | 동작 |
|------|-----------|------|
| Next.js | `next.config.*` 존재 | `npm run dev` → 자동 포트 → Chrome 열기 |
| HTML/정적 | `index.html` 존재 | `python3 -m http.server {port}` → Chrome 열기 |
| Python 스크립트 | `*.py` (웹 아님) | 실행 결과 HTML 리포트 변환 → Chrome 열기 |
| 기타 | 위에 해당 없음 | 터미널 출력 + 리포트 HTML 변환 |

### 포트 관리

- `lsof -i -P` 로 사용 중 포트 확인
- 3000~9999 범위에서 비어있는 포트 자동 선택
- 백그라운드 실행 (세션 종료 후에도 유지)
- 띄운 뒤 URL을 터미널에 표시 (iMessage 연결 시 iMessage에도 전송)

### 서버 생명주기

- 시작: "데모 띄워" 또는 워게임 PASS 후 자동
- 종료: "데모 꺼" 명령 또는 다음 워게임 시 이전 서버 자동 정리
- PID 파일: `reports/wargame/.demo.pid` 로 추적

## 4. 실수 패턴 학습

### 저장소

`reports/wargame/wargame-lessons.md`

> `rules/` 디렉토리가 아닌 `reports/wargame/`에 위치. 자율 모드(Step 4-A)에서 rules/는 수정 금지 대상이므로 충돌 방지.

### 구조

```markdown
# Wargame Lessons — 반복 실수 패턴 DB

> 워게임 루프에서 자동 누적. 수동 편집 가능.

## 반복 패턴

### [5회] 미사용 import 남김
- 최근: 2026-04-06, gbp-dashboard
- 대응: Phase 2에서 unused import 우선 체크

### [3회] API 응답 타입 누락
- 최근: 2026-04-05, company-web
- 대응: 타입 완전성 검증 강화

### [2회] .env.example 동기화 누락
- 최근: 2026-04-04, content-engine
- 대응: 빌드 전 .env/.env.example 비교
```

### 패턴 매칭 기준

오류를 **카테고리**로 분류하여 동일 패턴을 판별:

| 카테고리 | 매칭 기준 | 예시 |
|----------|-----------|------|
| `unused-import` | 미사용 import/변수 | JS/TS/Python import 정리 안 됨 |
| `type-missing` | 타입 선언 누락 | API 응답, 함수 파라미터 타입 없음 |
| `env-sync` | 환경변수 불일치 | .env ↔ .env.example 차이 |
| `xss-risk` | XSS 취약점 | innerHTML, dangerouslySetInnerHTML |
| `auth-missing` | 인증/인가 누락 | API 엔드포인트 보호 안 됨 |
| `e2e-selector` | E2E 셀렉터 깨짐 | 클래스명/ID 변경 후 테스트 미수정 |
| `build-dep` | 의존성 문제 | 패키지 미설치, 버전 충돌 |
| `기타` | 위에 해당 없음 | 오류 메시지 기반 자유 분류 |

프로젝트가 달라도 같은 카테고리면 동일 패턴으로 카운트.

### 학습 흐름

```
워게임 종료
  ↓
과거 리포트 스캔 (reports/wargame/*.html)
  ↓
현재 오류 카테고리 ↔ 과거 오류 카테고리 매칭
  ↓
2회 이상 반복 → wargame-lessons.md에 등록/카운트 증가
  ↓
다음 워게임 Phase 2에서 반복 패턴 우선 검사
  ↓
패턴 충분히 쌓이면 → 구현 단계 규칙으로 승격 제안
  (예: "이 실수 10회 넘었어요. rules에 추가할까요?")
```

### 규칙 승격 기준

- 동일 패턴 10회 이상 → "rules/에 추가할까요?" 사용자에게 제안
- 승격 시 해당 규칙은 구현 단계에서부터 적용 (검수 전에 예방)

## 6. 제약사항 및 리스크

| 리스크 | 대응 |
|--------|------|
| 세션 끊김 시 루프 중단 | 라운드별 상태 JSON 저장 (`reports/wargame/.state.json`: 현재 라운드, 잔여 이슈, 수정 이력). "이어서 해"로 재개 시 상태 파일 로드 |
| E2E 테스트 없는 프로젝트 | Phase 3 스킵, Phase 1+2만으로 검수 |
| 수정이 새 버그를 만드는 경우 | 연속 2라운드 같은 오류 시 자동 수정 중단 + 사용자 보고 |
| 리포트 파일 누적 | 워게임 시작 시 30일 이상 된 리포트 감지 → "오래된 리포트 N개 정리할까요?" 1회 제안 |
| API 토큰 소모 | Phase 2 병렬 실행으로 최소화, 라운드당 서브에이전트 3~4개 |
