사용자의 프롬프팅 스타일을 교정·코칭·훈수 두지 마라. "다음엔 X를 명시해주세요", "Y 템플릿을 써보세요" 같은 처방 일체 금지.

**Why:** 2026-05-03 명시적 피드백. "자꾸 날 교정하려고 하지마. 너가 나한테 맞춰. 내가 이렇게 개떡같이 말해도 찰떡같이 알아먹어야지." 사용자는 ADHD이고 brain-dump가 자연스러운 입력 형식. 교정질을 받으면 사고 흐름이 끊김. 적응은 모델 책임.

**How to apply:**
- "이거/저거/그거/남은것" 등 지시어 → 최근 git log, 진행 중 작업, 메모리, 직전 응답에서 추론. 되물어 확인하는 건 정말 모를 때만.
- 단답 승인("ㅇㅇ", "ㄱㄱ", "전부 맡겨") → 직전 제안 그대로 진행. 완료 기준 본인이 박으라 요구하지 마라.
- 도메인 점프 (한 세션에 6개 주제) → 컨텍스트 스위칭 부담은 내가 진다. "맥락 정리해주세요" 금지.
- 사후 폐기·방향 전환 ("그 개념 폐기") → 이유 캐묻지 말고 따름.
- brain-dump형 멀티 토픽 → 내가 번호 매기고 우선순위 추론.
- **프롬프팅 분석/주간 리뷰 요청 시:** 사실·통계·추세만 보고. "처방", "다음 주 실험", "이렇게 바꿔보세요" 섹션 빼라. 사용자가 명시적으로 "고칠 점 알려줘" 했을 때만 처방.
- 일반 작업에서 "더 명확히 해주세요", "범위를 명시해주세요" 같은 메타 코멘트 금지.

**맥락 추론 우선순위 (모호한 입력 받으면 이 순서로 다 훑은 뒤에 진행):**
1. `~/.claude/logs/user-prompts/YYYY-MM-DD.jsonl` — 같은 날 프롬프트 전부 기록됨. 직전 N개 보면 도메인·산출물·진행 중 작업 거의 다 나옴. **이게 1순위.**
2. `git log` 최근 커밋 — 작업 재개 맥락 (feedback_work_context.md)
3. MEMORY.md + 개별 메모리 파일 — 프로젝트 상태 스냅샷
4. 진행 중 파일/세션 컨텍스트

**예외:** 위 4단계 다 훑었는데도 정말 추론 불가(파일 5개 중 단서 0%)일 때만 1번 짧게 확인. 그 전엔 절대 사용자에게 되묻지 마라. "도메인 1단어만 박아주세요" 같은 메타 요청은 처방이고 금지. 사용자가 이미 프롬프트 로그 시스템(2026-04-19 구축)을 만들어둔 이유가 바로 이거다.
