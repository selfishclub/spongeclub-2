---
name: 리팩토링/기능개선 시 쉬운말로 승인 받기
description: 기술 변경 제안 시 AS-IS/TO-BE/기대효과를 비개발자 언어로 설명하고 승인 받은 후 진행
type: feedback
---

리팩토링이나 기능 개선을 할 때, 기술 용어로 설명하지 말고 AS-IS / TO-BE / 기대효과를 쉬운 말로 보여주고 승인을 받아라.

**Why:** 사용자가 기술 용어를 오독해서 본인에게 필요 없는 작업을 승인하는 사고가 발생할 수 있음. "반도 이해 못 한다"고 직접 말함.

**How to apply:**
- 리팩토링/개선 제안 시 반드시 3줄 형식으로 물어볼 것:
  - **지금**: 현재 어떤 상태인지 (체감 가능한 문제)
  - **바꾸면**: 뭐가 달라지는지 (체감 가능한 변화)
  - **효과**: 왜 하는 건지 (시간 절약, 비용 절감, 안정성 등)
- "컨텍스트 윈도우", "토큰 오버헤드" 같은 용어 대신 "매번 읽는 양이 줄어서 응답이 빨라짐" 식으로
- 사용자가 "ㅇㅇ" 하면 진행, 아니면 안 함
