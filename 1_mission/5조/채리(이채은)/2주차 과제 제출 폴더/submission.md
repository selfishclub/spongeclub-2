## 🎯 미션 1. 내 OS 만들기
> **[ 내 삶을 돕는 OS ]** 또는 **[ 콘텐츠 OS ]** 중 하나를 선택해 완성해주세요.

**✅ 선택:** 내 삶을 돕는 OS — (일/팀 영역) **팀 협업 허브 → 팀 운영체제(OS)**

> 사실 이건 1주차 인터뷰 때 제일 먼저 떠올렸던 아이디어였다. 그땐 다이어트 코치로
> 방향을 틀었지만, 이번 주에 그 "팀이 한 화면을 보며 움직이는 OS"를 실제로 만들었다.
> 그리고 하루 만에 v2 → v2.4까지 5개 버전으로 진화시켰다.

### 📐 기획
> 무엇을, 왜, 어떻게 만들지

**배경:** 우리 팀은 성수동 공간 대관 사업을 시작했다. **재무·운영·마케팅·BX·콘텐츠**
5개 파트로 역할을 나눠 움직이는데, 역할이 갈리다 보니 "다 같이 한 방향으로 가고 있는지,
서로 어디까지 진행됐는지"를 확인할 **공통 기준점**이 없었다. 그래서 팀 허브 홈페이지를
만들기로 했다. (공간 확보 완료, 8월 초 오픈 준비 중)

**방식 — 바로 만들지 않고 '방향 정렬 인터뷰'부터:**
Claude가 질문을 하나씩 던지고 내가 고르는 식으로 요구사항을 확정했다.
(사이트 성격 → 내부 허브로 시작 후 고객용 확장 / 담을 것 → 북극성·현황판·일정·자료)

### ⚙️ 구현
> 실제로 만든 것 (링크·스크린샷 — 이미지는 `이미지첨부/` 폴더에)

**① 기존 사이트 진단부터 (전환점이 된 발견)**
이미 만들어둔 홈페이지가 있었는데, Claude가 코드를 열어 진단했더니 —
모든 수정(체크박스·진행률·메모)이 **각자 브라우저에만 저장(localStorage)** 되고 있었다.
내가 "운영 80% 완료"를 체크해도 **팀원 화면엔 0%로** 보이는 구조. "팀이 같은 상황판을
본다"는 전제 자체가 안 서 있었다. ("팀원들이 잘 쓰고 있는지 확신이 없다"던 느낌의 진짜 원인)

**② 접근 방식 3안 중 선택**
A안(파일 기반 재구축) / B안(최소 수정, 기각) / C안(실시간 연동) 중
**C안 도전 + 실패 시 A안 전환**을 결정. (구조가 같아 전환 비용이 낮음)

**③ 설계안(화면 목업)으로 확정 → 두 가지 추가**
- **D-day**: 오픈일 기준 카운트다운 / **블로커**: 막힌 일이 홈 최상단에 자동 노출
- "세부는 노션" 컨셉을 폐기하고 **홈페이지를 팀의 1차 도구**로 정함

**④ 하루 동안의 버전 진화 (v2 → v2.4)** — 팀원들의 실제 요구가 그대로 다음 버전이 됐다

| 버전 | 무엇이 생겼나 |
|---|---|
| **v2** | 전 팀원이 같은 화면을 보는 상황판 (구글 시트 연동으로 localStorage 결함 해결) |
| **v2.1** | "웹에서 직접 못 고쳐?" → 구글 **Apps Script를 무료 백엔드**로 붙여 사이트에서 직접 추가·수정·삭제 (읽기 전용 → **양방향**) |
| **v2.2** | 북극성·가이드·자료실·공간 문서까지 **모든 텍스트를 팀원이 편집** 가능 |
| **v2.3** | 홈에 **자산 바로가기 타일**(시트·노션·인스타·블로그 등) + **📷 사진 탭**(공간 사진 21장, 원본 350MB→3.2MB 자동 변환) |
| **v2.4** | **한국어 스마트 검색** — 동의어(가격→대관료)·조사 무시("주차는 돼?"→주차)·초성(ㄷㄱㄹ→대관료)·희소성 랭킹 |

**⑤ 최종 사이트 — 6탭**
1. 🎯 홈 — D-day 배지 · 🚧 블로커 배너 · 북극성 카드 · 자산 바로가기 타일 · 그라운드 룰
2. 🧭 파트별 가이드 — 5개 파트(재무 신설), 전체 필드 편집
3. 📊 진행 현황 — 파트별 할 일·상태·담당·마감일·진행률, 추가/수정/삭제
4. 🗓 일정 — 날짜순, 항목별 D-day, 지난 일정 흐리게
5. 📷 사진 — 21장 그룹 갤러리
6. 📚 자료실 — 자료 CRUD + 공간 분석 문서(행 단위 편집)

**핵심 아키텍처 — "전 팀원이 같은 화면을 보는" 양방향 구조:**
```
팀원 → 홈페이지에서 직접 추가·수정·삭제 (또는 시트에서 직접)
          ↓ Apps Script API (무료, 구글 서버)
     구글 시트 (데이터 원본 · 자동 백업 · 버전 기록)
          ↓ 누가 언제 열어도
     전 팀원이 동일한 최신 상황판
```
**기술 스택:** 순수 HTML/CSS/JS (프레임워크·유료 서비스 0개) + Google Apps Script + Netlify.
데이터가 시트에 살아서, 나중에 다른 도구로 갈아타도 손실 없음.

### 🧗 과정에서의 삽질
> 막혔던 지점, 시도한 방법, 어떻게 풀었는지 솔직하게

- **가장 큰 삽질(이자 수확):** 겉보기엔 멀쩡한 기존 사이트가 사실 구조적으로 고장나
  있었다(localStorage). 감으로 "고칠까 새로 만들까" 정하려다, 코드를 직접 진단하게 해서
  원인을 찾은 게 프로젝트의 전환점이 됐다.
- **검색이 센스가 없던 문제:** v2.4에서 "공간 평수"를 검색해도 잘 안 나왔다. 원인은
  흔한 단어에 점수가 쏠린 것 → **희소성 가중치**(드문 단어를 무겁게) + 다중 단어 우선
  랭킹으로 고쳐서, 실제 데이터 12개 질의 전부 정답 1위로 만들었다.
- **실패를 미리 설계에 넣음:** C안(연동)이 안 될 때를 대비해 전환 기준과 fallback을
  먼저 정해서, 시트가 죽어도 사이트는 마지막 저장본으로 살아있게 했다.

### ✅ 결과물
> 완성한 것 / 작동 화면

- **6탭 팀 협업 OS 완성** — 양방향 직접 편집 + D-day + 블로커 + 자산 허브 + 사진 + 스마트 검색
- **품질 검증까지 마무리** ("만들었다"가 아니라 "동작함을 확인했다"):
  - 로직 테스트 **23개** 통과 (CSV 파서·D-day·블로커·진행률·검색까지 테스트 먼저 = TDD)
  - 헤드리스 브라우저로 3개 모드(읽기 전용/편집/API 장애) 렌더링 자동 검증
  - **API 왕복(E2E)** — 추가→수정→삭제가 실제 시트에 반영되는 것까지 코드로 확인
  - 연동을 일부러 끊어 경고 배지 + fallback 정상 동작 확인
  - 검색 품질 시뮬레이션 12종 전부 정답 1위
- **링크 / 스크린샷:** (내부용 사이트라 공개 여부 결정 후 첨부)

### 💡 알게 된 인사이트 & 공유하고 싶은 내용
> 하면서 깨달은 것, 크루들과 나누고 싶은 것

- **만들기 전에 진단부터.** "개선이냐 재구축이냐"를 감이 아니라 코드 레벨 진단으로 결정.
  겉보기 멀쩡한 사이트의 숨은 결함(localStorage)을 찾은 게 전부의 시작이었다.
- **질문 → 선택지 → 결정.** 요구사항을 한 번에 쏟지 않고 하나씩 확정하니 방향이 안 흔들렸다.
- **눈으로 보는 설계안.** 코드 대신 화면 목업으로 검토하니 비개발자인 나도 정확히 피드백.
- **아키텍처는 질문에서 진화한다.** "직접 수정 안 돼?" → Apps Script 백엔드, "검색 센스있게"
  → 희소성 랭킹. 내(사용자)의 불만이 가장 좋은 설계 입력이었다.
- **실패를 설계에 포함.** 전환 기준·fallback을 미리 정해두니 과감하게 시도할 수 있었다.
- **테스트가 속도를 만든다.** 하루에 5개 버전을 진화시킬 수 있었던 건, 테스트 23개와
  헤드리스 검증이 매 단계 "안 깨졌음"을 보장해준 덕분.
- **도구보다 의식(ritual), 완전함보다 독점.** 팀 도구의 성패는 기능이 아니라 "매주 열
  수밖에 없는 이유"에서 갈린다 — 그래서 현황·블로커·일정을 "오직 여기에만" 두기로 했다.
