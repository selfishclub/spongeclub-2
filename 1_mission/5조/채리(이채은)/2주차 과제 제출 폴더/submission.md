## 🎯 미션 1. 내 OS 만들기
> **[ 내 삶을 돕는 OS ]** 또는 **[ 콘텐츠 OS ]** 중 하나를 선택해 완성해주세요.

**✅ 선택:** 내 삶을 돕는 OS — (일/팀 영역) **팀 협업 허브 홈페이지**

> 사실 이건 1주차 인터뷰 때 제일 먼저 떠올렸던 아이디어였다. 그땐 다이어트 코치로
> 방향을 틀었지만, 이번 주에 그 "팀이 한 화면을 보며 움직이는 OS"를 실제로 만들었다.

### 📐 기획
> 무엇을, 왜, 어떻게 만들지

**배경:** 우리 팀은 성수동 공간 대관 사업을 시작했다. **재무·운영·마케팅·BX·콘텐츠**
5개 파트로 역할을 나눠 움직이는데, 역할이 갈리다 보니 "다 같이 한 방향으로 가고 있는지,
서로 어디까지 진행됐는지"를 확인할 **공통 기준점**이 없었다. 그래서 팀 허브 홈페이지를
만들기로 했다. (공간은 확보 완료, 8월 초 오픈 준비 중)

**방식 — 바로 만들지 않고 '방향 정렬 인터뷰'부터:**
Claude가 질문을 하나씩 던지고 내가 고르는 식으로 요구사항을 확정했다.

| 질문 | 나의 결정 |
|---|---|
| 사이트 성격은? | 내부 팀 허브로 시작 → 나중에 고객용 공개 페이지로 확장 |
| 사업 단계는? | 공간 확보 완료, 오픈 준비 중 |
| 팀원 업데이트 방식은? | 진행 상황은 팀원이 직접, 문서·자료는 내가 취합해 반영 |
| 사이트에 담을 것은? | 사업 방향(북극성) · 파트별 현황판 · 일정 · 공간 정보/자료 |

### ⚙️ 구현
> 실제로 만든 것 (링크·스크린샷 — 이미지는 `이미지첨부/` 폴더에)

**① 기존 사이트 진단부터 (전환점이 된 발견)**
사실 이미 만들어둔 홈페이지가 있었다. Claude가 코드를 직접 열어 진단했더니 —
모든 수정(체크박스·진행률·메모)이 **각자 브라우저에만 저장(localStorage)** 되고 있었다.
내가 "운영 80% 완료"를 체크해도 **팀원 화면엔 0%로** 보이는 구조. "팀이 같은 상황판을
본다"는 전제 자체가 안 서 있었다. ("팀원들이 잘 쓰고 있는지 확신이 없다"던 느낌의 진짜 원인)

**② 접근 방식 3안 중 선택**
- A안: 구조는 살리고 데이터를 파일로 재구축
- B안: 최소 수정 (핵심 문제 미해결 → 기각)
- C안: 구글 시트 연동 실시간 협업형
- **결정: C안 먼저 도전, 2주 안에 안 되면 A안으로 전환** (구조가 같아 전환 비용 낮음)

**③ 설계안(화면 목업)으로 확정 → 두 가지 추가 요청**
코드가 아니라 브라우저에서 보는 설계안으로 검토하니 정확히 피드백할 수 있었다.
- **D-day 기능**: 오픈일 기준 카운트다운
- **블로커 기능**: 막힌 일이 팀 전체에 즉시 보이게
- 또 "세부는 노션" 컨셉을 폐기하고 **홈페이지를 팀의 1차 도구**로 정함
  (가시화할 수 있는 건 전부 홈페이지에, 노션은 민감 자료 보관용으로만)

**④ 완성된 사이트 — 5개 탭 (기존 3개에서 확장)**
1. **🎯 홈** — D-day 배지 + 🚧 블로커 배너 + 북극성 카드(팀 목표·포지셔닝) + 그라운드 룰
2. **🧭 파트별 가이드** — 재무 추가로 5개 파트. 파트별 요약/목표/가이드/자료 링크
3. **📊 진행 현황** — 파트별 할 일·상태·담당·진행률 바
4. **🗓 일정** — 날짜순 마일스톤, 각 일정까지 남은 날, 지난 일정은 흐리게
5. **📚 자료실** — 공간 분석·사례 조사 자료 (민감 자료는 존재만 표시)

**핵심 아키텍처 — "전 팀원이 같은 화면을 보는" 구조:**
```
팀원 전원 → 공유 구글 시트에 입력 (현황/일정/공지)
              ↓ 사이트가 열릴 때마다 자동으로 읽음
          전 팀원이 동일한 최신 상황판
```
- 진행 상황·일정·공지 = 구글 시트가 원본 (팀원은 엑셀 쓰듯 입력만)
- 시트 상태를 "막힘"으로 바꾸면 → 홈 최상단 블로커 배너에 자동 노출
- 시트 연결이 실패해도 → 마지막 저장본으로 표시 + 경고 배지 (사이트가 죽지 않음)

**기술 스택:** 순수 HTML/CSS/JS (프레임워크·의존성 0개), 구글 시트 CSV 연동, Netlify 배포

### 🧗 과정에서의 삽질
> 막혔던 지점, 시도한 방법, 어떻게 풀었는지 솔직하게

- **가장 큰 삽질(이자 수확):** 겉보기엔 멀쩡한 기존 사이트가 사실은 구조적으로
  고장나 있었다(localStorage). "고칠까 새로 만들까"를 감으로 정하려다, Claude에게
  코드를 직접 진단하게 해서 원인을 찾은 게 프로젝트의 전환점이 됐다.
- **실패를 미리 설계에 넣음:** C안(시트 연동)이 안 될 수도 있어서, "2주 안에 안 되면
  A안으로" 기준을 먼저 정하고, 시트가 죽어도 사이트는 살아있는 fallback 구조를 짰다.

### ✅ 결과물
> 완성한 것 / 작동 화면

- **5탭 팀 협업 허브 완성** — 시트 연동 실시간 협업 + D-day + 블로커 + 5개 파트 현황판
- **품질 검증까지 마무리** ("만들었다"가 아니라 "동작함을 확인했다"):
  - 로직 테스트 13개 통과 (CSV 파서·D-day 계산·블로커 필터 등, 테스트 먼저 작성 = TDD)
  - 헤드리스 브라우저로 실제 렌더링해 13개 항목 자동 확인
  - 시트 주소를 일부러 틀리게 넣어 경고 배지 + fallback 정상 동작까지 검증
- **링크 / 스크린샷:** (아래 '확인 필요' 참고 — 내부용 사이트라 공개 여부 결정 후 첨부)

### 💡 알게 된 인사이트 & 공유하고 싶은 내용
> 하면서 깨달은 것, 크루들과 나누고 싶은 것

- **만들기 전에 진단부터.** "개선이냐 재구축이냐"를 감이 아니라 코드 레벨 진단으로 결정했다.
  겉보기 멀쩡한 사이트의 숨은 결함을 찾은 게 전부의 시작이었다.
- **질문 → 선택지 → 결정의 인터뷰 방식.** 요구사항을 한 번에 쏟지 않고 하나씩 확정하니
  방향이 안 흔들렸다. (1주차 인터뷰 스킬의 힘을 실전에서 또 느낌)
- **눈으로 보는 설계안.** 코드 대신 화면 목업으로 검토하니 비개발자인 나도 정확히
  피드백할 수 있었다 — D-day·블로커 추가, 노션 컨셉 폐기가 다 이 단계에서 나왔다.
- **실패를 설계에 포함.** 안 될 경우의 전환 기준과 fallback을 미리 정해두니 마음이 편했다.
- **TDD + 자동 검증.** AI가 "다 됐어요" 하는 걸 그대로 믿지 않고, 테스트와 헤드리스
  브라우저로 실제 동작을 확인하고 넘어가는 습관이 결과물의 신뢰도를 확 올렸다.
