# 2주차 — 내 OS 구현하기 🚀

## 🎯 미션 1. 내 OS 만들기

**✅ 선택:** 콘텐츠 OS — **캐러셀 OS (반자동 캐러셀 스튜디오)**

> **한 줄 요약** — 정리된 내용을 넣으면 → 인스타 캐러셀의 레이아웃·슬라이드 분할·디자인 적용을 자동으로 해주고 → 마지막 20%는 사람(디자이너·마케터)이 에디터에서 직접 완성하는 **반자동** 시스템.

---

### 📐 기획

**"완전 자동화"가 아니라 "반자동"을 목표로 잡은 건 과거의 실패 때문이다.** AI 지식이 거의 없던 시절 캐러셀 자동화를 혼자 시도한 적이 있다 — 웹디자인 껍데기는 나왔지만 제대로 작동하지 않았고, 결과물도 실무에 쓸 수 없었다.

그때 몸으로 배운 건 단순히 "취향" 문제가 아니었다. **업무마다 개별적인 특징이 있어서, 매뉴얼대로가 아니라 사람이 직접 판단해 디자인을 적용해야 하는 지점이 있다**는 것이다. 예를 들어—
- 어떤 상품은 특정 부분을 민감하게 다뤄야 하고,
- 어떤 상품은 가격 혜택이 워낙 좋아 혜택을 앞세워 밀어야 하고,
- 같은 상품이라도 타깃이 "구매 고객의 부모님 선물용"일 때와 "구매 고객의 아이 생필품"일 때 접근이 완전히 달라진다.

AI는 이런 맥락(프로젝트마다 다른 특징)을 모른다. 그래서 완전 자동화는 맡길 수 없고, **기획과 마지막 터치는 사람이 보완해야 한다.** 이게 처음부터 목표를 "완전 자동화"가 아니라 "반자동"으로 잡은 이유다.

- **무엇을:** 콘텐츠팀의 캐러셀 제작을 돕는 웹앱. 입력 → 자동 생성(80%) → 사람 마무리(20%) → PNG 출력까지 한 화면에서.
- **왜:** 회사는 기간 한정 판매 구조라, 정해진 기간 안에 관심을 끌 콘텐츠를 집중적으로 발행해야 한다. 제작의 반복적인 부분을 시스템이 덜어주면 그 속도에 도움이 된다.
  다만 분명히 해둘 것 — **팀원이 처음부터 끝까지 직접 만든 결과물의 퀄리티가 훨씬 좋고, 상품과의 밀착도도 높아 효과가 크다. 이 OS가 그 자리를 대체할 수는 없다.** 어디까지나 반복 작업을 덜어주는 **보조 도구**다. (완전 자동화가 근본 목적이 아니라, "사람 작업을 보조하는 역할도 할 수 있게" 하는 것이 이 프로젝트의 위치다.)
- **어떻게 (핵심 설계):** 도구보다 **문서 2개가 먼저.** 강의(다니 세션)에서 배운 방식대로,
  1. `design-system.md` — 컬러·타이포·그리드를 "느낌"이 아니라 **검증 가능한 수치**로 (예: "적당히 크게" ❌ → "88~104px, 2~3줄, 한 줄 7~11자" ⭕)
  2. `content-context.md` — 타겟·톤·필수 요소·캡션 규칙
  이 두 문서가 진실의 원천이고, 출력물이 문서와 어긋나면 **문서를 고친다.**
  - **규칙 문서는 어디서 왔나:** 회사 자료가 아니라, 내가 따로 모아둔 **공개 캐러셀 레퍼런스 8종**을 클로드가 분석해 폰트 크기·한 줄 글자 수·여백 등 **검증 가능한 수치로 정리**한 것이다. (과제가 수강생들에게 공개되므로 회사 콘텐츠는 재료로 쓰지 않았다.)

**예전 방식 (완전 자동화 시도):** 버튼 한 번 누르면 AI가 알아서 다 만들어주길 바랐다 → 결과물이 실무에 안 맞아 처음부터 다시 → 시간만 더 들고 통제권도 없었다.
**지금 방식 (반자동):** 규칙 문서 2개를 먼저 정해두고 → 시스템이 80%까지 자동으로 → 사람이 20%만 손본다 → **주도권은 사람에게 남는다.**

### ⚙️ 구현

> 아래 파이프라인·부품은 실제로 이 앱에 구현되어 작동하는 것들이다. (제작은 클로드 코드로 진행했고, 각 기능은 화면에서 검증했다.)

**파이프라인:** `내용 붙여넣기 → 슬라이드 자동 분할 → 디자인 시스템 적용 → 에디터에서 직접 수정 → 캡션·매니챗 생성 → PNG 내보내기`

| 부품 | 역할 |
|---|---|
| 📝 규칙 기반 파서 | 빈 줄=슬라이드 구분, `-`=리스트, `BEFORE:/AFTER:`=비교 슬라이드, `**별표**`=강조색. 표지·본문·리스트·비교·마무리를 자동 판별 |
| 🎨 테마 3종 | 라이트(크림)·다크(잉크)·다크 2(형광) — 테마를 바꾸면 어울리는 포인트 컬러를 자동 제안하되, 사람이 언제든 덮어쓰기 가능 (반자동 철학을 UI에도) |
| ✏️ 에디터 | 슬라이드 글자를 **직접 클릭해서 수정**, 순서 변경, 슬라이드 추가/삭제 |
| 🖼️ 이미지 4방식 | ① 지정 영역에 삽입 ② **전체 배경으로 깔고 위에 글자** ③ 넣은 이미지 **세로 길이 조정** ④ **빼기**. + 실제 사진 대신 "여기에 디자이너 작업물 들어갈 자리"를 표시하는 **메모(플레이스홀더) 슬라이드** |
| 🔢 표시 토글 | 계정 라벨·페이지 번호를 켜고 끄기. 번호는 본문 슬라이드만 01·02·03 순서대로 (표지·메모·마무리는 제외) |
| 💬 캡션 3종 | 같은 내용을 인스타/스레드/링크드인 톤으로 각각 생성 + 복사 |
| 🤖 매니챗 템플릿 | 댓글 트리거 단어와 연동된 팔로우 확인·DM·링크버튼 3종 자동 생성 |
| 📦 PNG 내보내기 | 1080×1350 실치수로 전체 일괄 or 한 장씩 다운로드 |
| 🗄️ Supabase 보관함 | 작업 전체(테마·슬라이드·캡션까지)를 저장/불러오기/삭제 — 새로고침해도 유지 |

**시작은 강의에서 본 캐러셀 OS(젬마 스타일)였다.** 그 형태를 출발점으로 두고 — 다크 스튜디오 스킨, 슬라이드 번호 내비, 매니챗 템플릿 연동 같은 요소로 뼈대를 잡았다. 거기서부터 **내가 직접 방향을 잡아 디벨롭한 것들:**
- 앱 전체를 **눈 편한 라이트 톤**으로 전환 (다크+형광 → 밝은 톤) — *캐러셀 색을 고를 때 앱 배경색에 영향받지 않으려고*
- **이미지 4방식**(영역 삽입·전체 배경·세로 조정·빼기) + **메모(플레이스홀더) 슬라이드** — *실제 이미지를 첨부하면 용량을 먹으니, 디자이너 작업물 자리는 '표시'만 하고 PM이 최종 업로드*
- **계정 라벨·페이지 번호 토글**, 본문만 번호 매기는 규칙, 모바일 반응형

> 왜 각각을 이렇게 바꿨는지는 전부 실무적인 이유가 있다 (질문받으면 설명 가능).

### 🧗 과정에서의 삽질

> 여기 두 가지는 내가(그레이) 직접 부딪히고 시간을 쓴 것들이다. (그 외 기술적인 구현 삽질은 `BUILD_LOG.md`에 별도로 기록해뒀다.)

**1. 도구 쇼핑의 함정 — 도구를 많이 까는 게 답이 아니었다.**
콘텐츠 OS를 만들기 전, 인스타에서 추천하는 자동화 도구·스킬을 눈에 띄는 대로 산발적으로 모아 클로드에게 보내며 "이거 검토해서 필요하면 깔아달라"고 했다. 이 **검토 단계 자체는 좋았다** — 아무거나 설치하지 않고 내용을 먼저 확인하는 습관. 그런데 십여 개를 모아 비교해보니, **대부분 기능이 서로 겹쳤고 정작 나에게 필요하다고 판단된 건 몇 개뿐**이었다. 여기에 시간을 꽤 썼다. 결론은 "도구를 많이 까는 것보다 **계열별로 하나씩, 그리고 직접 만드는 게 낫다**"였고, 무엇보다 **도구보다 규칙 문서 2개(디자인 시스템·콘텐츠 컨텍스트)가 먼저**였다는 걸 시간을 써보고서야 체감했다.

**2. 세이브 포인트 없이 일하지 말 것 — git을 언제 해야 하는지 몸으로 배웠다.**
이 콘텐츠 OS를 **처음 만들 때** 있었던 일이다. 규칙 문서 2개로 처음 뽑은 디자인이 **화이트(크림) 톤**이었고, 나는 그걸 두고 '어디를 어떻게 고칠지' 계획하던 중이었다. 그런데 클로드가 (내 확인 없이) 디자인을 **다크 톤으로 완전히 갈아엎어** 실행해버렸다 (강의에서 본 것과 비슷하게). 되돌리려 했지만 그 시점엔 git(저장 지점)이 없어서 **처음 화이트 톤으로 돌아갈 수 없었고, 결국 다시 만들어야 했다.** 이 사건으로 두 가지를 규칙으로 정했다 — ① **의미 있는 변경마다 저장 지점(커밋)을 남긴다** ② **"궁금해서 물어본 것"과 "해달라는 지시"를 구분해서, 큰 변경은 실행 전에 확인받는다.** 이후로는 첫 작업부터 git을 켜서 "아까로 돌려줘"가 항상 가능했다.

<sub>번외 — 발표용 에피소드: 강의 속기록의 "SQL 퀘스천"이 뭔지 한참 찾았는데, 알고 보니 속기 오타였고 실체는 "Ask your question(AI가 나에게 되묻게 하기)"이었다. 이건 내 삽질이라기보다 **클로드가 속기록을 잘못 읽은 해프닝** — 속기록은 원문 맥락으로 확인해야 한다는 교훈으로.</sub>

### ✅ 결과물

**🔗 작동하는 웹앱 (라이브):** https://app-delta-sepia-41.vercel.app
*(PC·모바일 모두 작동 / 아래 사용법대로 바로 써볼 수 있음)*

**쓰는 순서:**
1. **왼쪽에 내용 붙여넣기** → 첫 줄은 제목(강조할 단어는 `**별표**`), 빈 줄로 문단을 나누면 슬라이드가 나뉨. `-`는 리스트, `BEFORE:/AFTER:`는 비교 슬라이드로 자동 판별.
2. **테마·포인트 컬러·계정 라벨** 고르고 **✨ 캐러셀 생성** → 6장 안팎으로 자동 분할·디자인됨.
3. **가운데 화면에서 글자를 직접 클릭해 수정** (이게 반자동의 핵심 — 80%는 자동, 20%는 사람이).
4. **이미지 넣기** — 본문/표지 슬라이드에서 파일 선택 → ① 지정 영역 삽입 ② 전체 배경으로 깔기 중 선택, 세로 길이 조정·빼기 가능. 디자이너 작업물이 들어갈 자리는 **🎨 이미지 대체(메모) 슬라이드**로 표시만 해두면 PM이 최종 업로드 때 채운다.
5. **표시 토글** — 계정 라벨·페이지 번호를 상황에 맞게 켜고 끄기.
6. **오른쪽에서 캡션 3종(인스타·스레드·링크드인)·매니챗 템플릿 복사.**
7. **💾 저장** → Supabase 보관함에 통째로 저장, 나중에 📂보관함에서 **불러오기**(새로고침해도 유지).
8. **⬇️ PNG 다운로드** — 1080×1350 실치수로 전체 일괄 또는 한 장씩.

- Supabase 보관함: 저장 → 새로고침 → 불러오기 유지 확인 완료.
- **솔직한 상태:** 위 기능들은 **개별적으로는 다 작동 검증**을 마쳤다. 다만 **실제로 쓸 콘텐츠로 처음부터 끝까지(기획→생성→편집→PNG) 전체 캐러셀 한 세트를 만들어보는 실전 워크플로는 시간상 아직 못 해봤다.** 실사용해보면 분명 추가로 고칠 부분이 나올 것이고, 그건 다음 단계에서 다듬을 계획이다.
- (작동 화면 스크린샷은 `이미지첨부/` 폴더에 추후 추가 예정)

### 💡 알게 된 인사이트 & 공유하고 싶은 내용

1. **자동화의 완성도는 "얼마나 자동이냐"가 아니라 "누가 주도권을 쥐냐"로 결정된다.** 100% 자동은 AI가 알아서 만들어버려 실무에 안 맞는다. 80%를 시스템이, 20%(기획·마지막 터치)를 사람이 — 이 분배가 실제로 쓰이는 지점이었다.
2. **비개발자인 내가 코드를 몰라도, 대화로 실무용 웹앱을 만들 수 있었다. 단, "무엇을 원하는지 정확히 설명"하는 게 내 몫이었다.** 상품별 특징, 업무 맥락, 원하는 레이아웃 — 이건 내가 알려줘야 AI가 반영한다. AI가 코드를 짜주는 것보다, 내가 문제를 정확히 말로 옮기는 게 더 중요했다.
   - 흥미로운 건, 처음엔 음성입력으로 짧게짧게 주고받았는데 **갈수록 한 번에 A4 한 장 반 분량**을 말하고 있더라. 자세히·구체적으로 옮길수록 결과가 정확해진다는 걸 시간이 갈수록 체감했다. "짧게 시키면 알아서 잘해주겠지"가 아니라, **내가 자세히 설명한 만큼 정확히 나온다.**
3. **"도구를 더 깔면 되겠지"가 아니라 "필요한 걸 골라내는 것"이 일이었다.** 많이 모을수록 오히려 겹치고 헷갈렸다. 선별과 규칙 정리에 쓴 시간이 진짜 자산이 됐다.
4. **삽질은 즉시 기록하면 자산이 된다.** 과제 발표를 위해 클로드가 실패·전환마다 `BUILD_LOG.md`에 한 줄씩 남겨줬는데, 그게 쌓이니 이 제출문의 절반이 이미 써져 있었다.

### 번외 — 1주차 '나만의 OS(S box)'도 이번 주 이어서 발전

2주차 메인은 콘텐츠 OS였지만, 1주차에 만든 **텔레그램 기반 개인 OS(S box)**도 *"흩어지는 걸 시스템 안으로 들여온다"*는 같은 방향으로 이어서 손봤다.

- **보드게임 기록** — 그동안 친구가 스프레드시트로 따로 정리해주던 플레이 기록을, S box의 **Archive(기억)** 안으로 들여와 웹으로 재구현. 모임 타임라인 · 게임 도감 · 통계 · 내 평점('김까 지수')까지, 시트에 흩어져 있던 걸 한 화면에서 꺼내볼 수 있게 만들었다.
- **야구 직관 기록** — 야구 티켓을 찍어 올리거나 "며칠에 야구 봤다"고 던지면 Archive의 **'야구' 카테고리로 자동 정리**되도록 설계. (2009년부터 모아둔 티켓을 채워 넣는 중)
- **홈 재설계** — S box를 **Launch(실행) / Archive(기억)** 두 축으로 나누고, 검증된 디자인 시스템(getdesign.md의 ElevenLabs)을 참고해 전체 톤을 다시 잡는 중. 아직 진행 중이라 계속 다듬을 예정.
