# 2주차 — 내 OS 구현하기 🚀

## 🎯 미션 1. 내 OS 만들기

**✅ 선택:** 콘텐츠 OS — **캐러셀 OS (반자동 캐러셀 스튜디오)**

> **한 줄 요약** — 정리된 내용을 넣으면 → 인스타 캐러셀의 레이아웃·슬라이드 분할·디자인 적용을 자동으로 해주고 → 마지막 20%는 사람(디자이너·마케터)이 에디터에서 직접 완성하는 **반자동** 시스템.

---

### 📐 기획

**"완전 자동화"가 아니라 "반자동"을 목표로 잡은 건 과거의 실패 때문이다.** AI 지식이 거의 없던 시절 캐러셀 자동화를 혼자 시도한 적이 있다 — 웹디자인 껍데기는 나왔지만 제대로 작동하지 않았고, 결과물도 내 마음에 안 들었다. 그때 몸으로 배운 것: **클릭 한 번으로 다 해주는 자동화는 내 취향을 모른다.**

- **무엇을:** 콘텐츠팀의 캐러셀 제작을 돕는 웹앱. 입력 → 자동 생성(80%) → 사람 마무리(20%) → PNG 출력까지 한 화면에서.
- **왜:** 회사는 기간 한정 판매 구조라 그 기간 안에 콘텐츠를 집중 발행해야 한다. 팀원 공백이 생겨도 제작이 멈추지 않으려면, "만드는 과정"의 반복 부분을 시스템이 대신해줘야 한다.
- **어떻게 (핵심 설계):** 도구보다 **문서 2개가 먼저.** 강의(단이 세션)에서 배운 방식대로,
  1. `design-system.md` — 컬러·타이포·그리드를 "느낌"이 아니라 **검증 가능한 수치**로 (예: "적당히 크게" ❌ → "88~104px, 2~3줄, 한 줄 7~11자" ⭕)
  2. `content-context.md` — 타겟·톤·필수 요소·캡션 규칙
  이 두 문서가 진실의 원천이고, 출력물이 문서와 어긋나면 **문서를 고친다.**

**BEFORE:** 클릭 한 번 → 결과물 → 마음에 안 들어서 처음부터 다시
**AFTER:** 규칙 문서 2개 → 80% 자동 → 사람이 20%만 손보기

---

### ⚙️ 구현

**파이프라인:** `내용 붙여넣기 → 슬라이드 자동 분할 → 디자인 시스템 적용 → 에디터에서 직접 수정 → 캡션·매니챗 생성 → PNG 일괄 다운로드`

| 부품 | 역할 |
|---|---|
| 📝 규칙 기반 파서 | 빈 줄=슬라이드 구분, `-`=리스트, `BEFORE:/AFTER:`=비교 슬라이드, `**별표**`=강조색. 표지·본문·리스트·비교·CTA를 자동 판별 |
| 🎨 테마 3종 | 라이트(크림)·다크(잉크)·젬마(형광) — 테마를 바꾸면 어울리는 포인트 컬러를 자동 제안하되, 사람이 언제든 덮어쓰기 가능 (반자동 철학을 UI에도) |
| ✏️ 에디터 | 슬라이드 글자를 **직접 클릭해서 수정**, 순서 변경, 슬라이드 추가/삭제, 이미지 업로드 |
| 💬 캡션 3종 | 같은 내용을 인스타/스레드/링크드인 톤으로 각각 생성 + 복사 버튼 |
| 🤖 매니챗 템플릿 | 댓글 트리거 단어와 연동된 팔로우 확인·DM·링크버튼 3종 자동 생성 |
| 📦 PNG 내보내기 | 1080×1350 실치수로 전체 일괄 or 한 장씩 다운로드 |
| 🗄️ Supabase 보관함 | 작업 전체(테마·슬라이드·캡션까지)를 저장/불러오기/삭제 — 새로고침해도 유지 |

**벤치마크도 직접 했다.** 강의 데모(젬마 캐러셀)를 브라우저로 열어 기능 지도를 그려보니 — 데모엔 '내용 입력'과 '글자 편집'이 없었다. 우리 v0는 입력→자동생성이 있어서 오히려 한 발 앞서 있었고, 대신 데모의 좋은 점(다크 스튜디오 스킨, 슬라이드 번호 내비, 매니챗 연동)을 이식했다.

### 🧗 과정에서의 삽질

| 막힌 벽 | 어떻게 풀었나 |
|---|---|
| **CSS transform의 배신** — 슬라이드(1080×1350)를 화면에 맞춰 `scale()`로 축소했더니 "보이는 크기"만 줄고 "차지하는 자리"는 안 줄어서 레이아웃 붕괴 | 홀더 박스 크기를 축소 비율로 따로 계산해서 해결. **transform은 시각 효과일 뿐 레이아웃엔 영향이 없다**는 걸 배움 |
| **"SQL 퀘스천"이 뭐지?** — 강의 속기록에 나온 단어를 한참 검색 | 속기 오타였다. 실체는 **"Ask your question"** (AI가 나에게 되묻게 하기). 속기록은 원문 맥락으로 확인해야 한다 |
| **도구 쇼핑의 함정** — 인스타 추천 자동화 도구·스킬 13개를 수집해서 비교 | 대부분 서로 중복이었다. 결론: **계열별 1개 + 직접 만들기.** 문서 2개가 도구 10개보다 중요했다 |
| **열쇠가 다 같은 열쇠가 아니다** — "API 키는 무조건 숨겨야 한다"고만 알고 있었음 | Supabase엔 anon(브라우저 공개용 = 가게 앞문 열쇠)과 service_role(진짜 비밀 = 금고 열쇠)이 따로 있다. 키마다 용도가 다름 |
| **인플루언서 공유 스킬 저장소를 설치하려다 안전장치에 걸림** | 외부 코드는 내용물을 검사한 뒤 설치하기로. 좋은 습관이 하나 생겼다 |
| **(1주차 S box에서 이어진 교훈)** 크림색 디자인을 되돌리고 싶은데 세이브 포인트가 없어서 못 돌린 사건 | 이번 프로젝트는 **첫 작업이 git init** — 의미 있는 변경마다 커밋해서 언제든 "아까로 돌려줘"가 가능하게. 삽질도 즉시 BUILD_LOG.md에 기록 (지금 이 표가 그 파일에서 나옴) |

### 💡 알게 된 인사이트 & 공유하고 싶은 내용

1. **자동화의 완성도는 "얼마나 자동이냐"가 아니라 "누가 주도권을 쥐냐"로 결정된다.** 100% 자동은 AI 취향대로 나온다. 80%를 시스템이, 20%를 사람이 — 이 분배가 실무에서 진짜 쓰이는 지점이었다.
2. **문서 2개 > 도구 10개.** 레퍼런스 13개를 모아 비교하고 얻은 결론이 "디자인 시스템·콘텐츠 컨텍스트 문서로 AI와 싱크를 맞추는 것"이었다. 규칙을 수치로 적는 순간 AI가 내 규칙대로 일하기 시작한다.
3. **v0는 일부러 LLM 없이 규칙 기반 파서로 만들었다.** 단순하지만 예측 가능하다 = 반자동 철학과 일치. AI 글다듬기는 다음 버전에서 붙인다 (순서가 중요하다 — 뼈대가 예측 가능해야 AI를 얹어도 통제가 된다).
4. **삽질은 즉시 기록하면 자산이 된다.** BUILD_LOG.md에 실패할 때마다 한 줄씩 남겼더니, 이 제출문의 절반이 이미 써져 있었다.
