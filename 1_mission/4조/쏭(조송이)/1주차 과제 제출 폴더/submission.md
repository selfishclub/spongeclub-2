# 1주차 — 나만의 OS 만들기 🛠️

> 미션을 진행하며 **과정과 결과물**을 기록해주세요. (다 못 채워도 OK, 한 것 위주로!)

## 🎯 미션 1. 내 OS 재료 찾기
> 인터뷰 스킬(아이데이션)로 "내 삶에 필요한 게 뭔지" 찾기

- **과정 (어떻게 찾았나):**
  인터뷰 스킬을 따라 하루를 떠올리는 것부터 시작했다. 어제 하루는 *클로드랑 앱 만들기 / 일 / 집 청소* 로 채워졌다. 마찰점을 찾다 보니 두 개가 나왔다 — 삶 쪽은 "집 청소를 몰아서 대청소하지 말고 매일 조금씩 체크리스트로", 일 쪽은 "클로드와 장기 프로젝트하며 쌓인 md 파일을 정기적으로 정리·퍼지". 둘을 나란히 놓고 보니 **둘 다 '정기적으로 조금씩 봐줘야 하는 유지보수'라는 같은 모양**이라는 걸 알아챘다. 이번 주는 일 쪽(md 파일)을 시작점으로 골랐다.

- **결과 — 🟡 내 OS 재료 카드:**

  | 항목 | 내용 |
  |---|---|
  | **영역** | 일 (클로드와의 장기 프로젝트 운영) — "정기 유지보수"라는 같은 틀로 삶 쪽 '집 청소 데일리'에도 그대로 얹을 수 있음 |
  | **걸리는 지점** | 큰 chunk가 끝나도 md 파일이 최신화가 안 됨. 클로드가 기존 폴더 대신 새 md를 만들고, prompt·진행중 파일은 업데이트가 밀려 stale 정보가 쌓임 |
  | **지금은** | `session digest` 스킬이 트리거·정리·stale 리뷰까지는 해줌. 그런데 ① purge(덜어내기)가 빠져 있고 ② 결정은 내 몫인데 "정했을 때 뭘 해야 하는지"를 나도 클로드도 매번 기억 못 해서 gap이 남음 |
  | **OS가 된다면** | 결정은 내가(= "이번엔 비우자" 신호) → 그 신호가 오면 purge 모드가 켜져서, 내가 절차를 기억 안 해도 stale 파일을 지우거나 합쳐 책상을 비움. digest와 분리된 별도 주기 — 매번이 아니라 쌓였을 때만 |
  | **한 문장** | "내가 끝났다고 말하기만 하면, 나머지는 시스템이 알아서 비운다" |
  | **첫 한 걸음** | session digest와 분리된 'purge' 트리거 하나 만들기 — 신호를 주면 stale 후보를 모아 "지울까요 / 합칠까요?"까지 물어보고 실행하는 단계 |

- **느낀 점:**
  "딱히 안 불편한데"가 아니라 이미 굴리던 시스템(session digest)의 *빠진 한 조각*이 진짜 재료였다. 그리고 인터뷰 중에 "phase 전환점은 누가 정하나? 결국 나 아닌가?"라고 스스로 따져본 게 결정적이었다 — OS의 역할이 *결정 대신해주기*가 아니라 *결정한 뒤의 절차를 기억하고 실행하기*라는 걸로 정리됐다. 완벽하지 않아도 gap을 조금씩 줄여주면 그게 OS다.

## 🧩 미션 2. 내 OS 기획
> 인터뷰 결과 + 세션 내용(흐민·배짱·키노) 활용해 기획

- **세션 소스에서 가져온 것 (흐민·배짱·젬마 0628 1회차):**
  - **(젬마/하네스)** "**Claude.md가 너무 길어지면 오히려 AI가 내용을 누락한다**" + 하네스(Harness) 개념 — CLAUDE.md·지침·스킬로 AI가 "고삐 풀린 망아지"가 안 되게 잡는 것. → 세션이 *내 OS가 푸는 문제를 그대로 명명*해줬다: **하네스가 비대해지면 시스템이 망가진다.**
  - **(흐민 5단계 루프)** 인풋→저장→연결→제안→발행 = 지식을 *쌓는* 흐름. + **External/Self 폴더 분리**로 혼재 방지.
  - **(배짱 온맘)** "완성보다 뭐라도 하나", 작게 시작해 디벨롭, 텔레그램으로 자동화.

- **소스가 내 OS를 어떻게 다시 잡아줬나:**
  흐민의 루프가 지식을 *쌓는다면*, 내 purge는 그 쌓인 걸 *덜어내 하네스를 건강하게 유지*하는 **반대쪽 절반**이다. 둘이 같이 돌아야 OS가 안 터진다.
  - 쌓기(흐민): `인풋 → 저장 → 연결 → 제안 → 발행`
  - 덜기(나): `session digest → session review → purge` ← 내가 만든 조각

  그리고 흐민의 **External/Self 분리**는 내 **Raw/Refined 분리**와 같은 원리 — 혼재를 막아 시스템이 정확히 읽히게 한다. (Raw=보존, Refined=갱신)

- **기획 내용:**

  기획하며 카드보다 한 단계 더 또렷해졌다. 핵심은 **두 층 모델**이다.

  - **Raw (원본 기록)** — session digest, 아카이브용 요약. 무조건 보존, **불가침**. purge 대상이 아니다.
  - **Refined (정제된 현재상태)** — 규칙·현재 상황·어디까지 했나·결정 같은 *계속 갱신되는* 문서. purge가 사는 곳.

  그래서 purge가 하는 일은 "파일 지우기"가 아니라 **Refined 문서가 최신 결정과 어긋나지 않게 유지하는 것**이다. 새 결정이 옛 결정을 덮으면 이제 틀린 옛 서술을 들어낸다. 단, **충돌 시 무엇이 무엇을 override하는지는 그때 내가 판단** — 시스템이 자동으로 최신을 고르지 않고, 옛것↔새것을 나란히 띄워 묻는다.

  스코프가 둘이라 **스킬도 둘**로 나눴다 (위험 반경이 다르기 때문):
  - **`purge-project`** — 한 프로젝트 안의 Refined 문서 정리. 신호 받으면 그 순간 새로 스캔 → 충돌/낡음 후보 제시 → 항목별 확인 → 교체·제거.
  - **`purge-system`** — 전역 CLAUDE.md·memory·개인 스킬 검토. 반경이 커서 한 단계 더 보수적(미리보기 먼저, 백업 제안, 삭제보다 아카이브). 파이프라인 밖, 아주 가끔.

  전체 그림: `session digest → session review → purge-project` (Raw 쌓기 → stale 넛지 → Refined 최신화), 그리고 그 바깥에 독립적으로 `purge-system`.

- **막혔던 점 / 어떻게 풀었나:**
  1. **트리거 자동감지 → 사람 판단.** 처음엔 "phase 전환점을 시스템이 감지"하려 했는데, design phase는 끝이 모호하고 *그걸 정하는 건 결국 사람*이라 막혔다. → **"결정은 나 / 절차 기억·실행은 OS"** 로 역할을 나눠 풀었다.
  2. **단위를 잘못 잡음 (파일 → Raw/Refined).** "낡은 파일을 지운다"로 생각했는데, 진짜 단위는 *Raw(보존)냐 Refined(갱신)냐* 였다. Raw는 절대 안 건드리고, Refined의 *내용*만 override하는 것으로 정리하니 깔끔해졌다.
  3. **표준과의 충돌.** 0-Standard 문서를 리뷰하니 "낡은 건 삭제가 아니라 `0-Archive`로" "`project-log.md` 등 living docs는 보존" 규칙이 이미 있었다. → 스킬에 *Raw 불가침 목록*과 *아카이브 우선* 원칙을 반영해 표준에 맞췄다.
  4. **세션 원칙 적용 — "AI는 동조한다."** 젬마 세션의 "AI는 긍정 방향으로 학습돼 사용자에 동조하니 반대 관점을 요청하라"를 그대로 반영했다. purge가 충돌 시 자동으로 최신을 고르지 않고 **옛것↔새것을 나란히 띄워 사람이 판단**하게 한 게 이 원칙의 구현이다.

## ⚙️ 미션 3. 내 OS 구현
> 실제로 만들어본 것 (클로드코드 '채널' 기능 활용 OK)
- **결과물:** 개인 스킬 **두 개를 실제로 구현**해 클로드 코드에 등록 완료 (`~/.claude/skills/`).
  - `purge-project/SKILL.md` — 프로젝트 Refined 문서 최신화 스킬 (두 층 모델, 충돌 시 사람 판단, 새로 스캔→항목별 확인→교체·제거)
  - `purge-system/SKILL.md` — 전역 자산 보수적 검토 스킬 (CLAUDE.md·memory·스킬, 미리보기 먼저)
  - 두 스킬 모두 세션에서 정상 로드 확인. 표준 단일 진실원천인 `0-Standard/skill-directory.md`에도 등록(Wrapping Up + ground-truth)해 read-protocol과 정합성 유지.
- **링크 / 스크린샷:** (이미지는 `이미지첨부/` 폴더에) — 스킬 파일은 개인 환경(`~/.claude/skills/`)에 위치해 이 repo 밖에 있음.
